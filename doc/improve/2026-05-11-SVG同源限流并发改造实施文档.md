# AI-Passage-Creator SVG 同源限流并发改造实施文档

## 1. 目标

本文只讲一件事：当一次任务里有多张 SVG_DIAGRAM 图片时，如何把当前“同源串行”改成“同源限流并发”，从而压缩 SVG 总耗时。

目标不是重写整个配图链路，而是在尽量少改现有结构的前提下，把当前这条路径：

1. ParallelImageGenerator 按来源分组
2. SVG_DIAGRAM 这一组内部逐张串行
3. 每张 SVG 各自调用一次大模型

改成：

1. 仍然按来源分组
2. 只有 SVG_DIAGRAM 这一组放宽为有限并发
3. 其他来源保持现状不动
4. 单张图片完成时的 SSE 进度消息仍然保留
5. 最终图片结果仍按 position 排序

## 2. 当前代码锚点

这次改造只需要重点看 4 个文件：

- src/main/java/com/ywt/passage/agent/parallel/ParallelImageGenerator.java
- src/main/java/com/ywt/passage/config/AsyncConfig.java
- src/main/java/com/ywt/passage/config/SvgDiagramConfig.java
- src/main/resources/application.yaml

当前现状：

### 2.1 ParallelImageGenerator 当前策略

当前代码在 ParallelImageGenerator 里是：

- 先把 imageRequirements 按 imageSource 分组
- 不同来源用 CompletableFuture.runAsync 并行
- 但每个来源组内部仍然是 for 循环串行

也就是说，如果 5 张图全是 SVG_DIAGRAM，实际还是 5 次顺序执行。

### 2.2 当前 runAsync 使用的是默认线程池

当前代码里 source 分组任务是直接调用 CompletableFuture.runAsync，没有显式传 Executor。

这意味着：

- 目前使用的是默认公共线程池
- 后面如果继续往里面塞更多 SVG 子任务，线程行为会变得不可控
- 为了让“SVG 并发数 = 配置值”真正可控，最好补一个专用线程池

### 2.3 当前 SvgDiagramConfig 还没有并发配置

现在 SvgDiagramConfig 只有：

- defaultWidth
- defaultHeight
- folder

还没有 maxConcurrency 这种字段。

## 3. 推荐实现策略

推荐使用“专用线程池 + 只对 SVG 分支做有限并发”的实现方式。

不要采用下面两种做法：

- 不加限制，直接把同组所有 SVG 全量并发
- 继续使用 CompletableFuture 默认线程池，靠运气限制并发

推荐理由：

- 改动面小，只需要在现有 ParallelImageGenerator 里加分支
- 逻辑清晰，后续出问题也容易排查
- 并发上限可配置，方便后续根据模型限流情况调参
- 不会影响 PEXELS、EMOJI_PACK、ICONIFY、MERMAID 的现有行为

## 4. 改动总览

这次改造建议按下面顺序做：

### 第 1 步：给 SvgDiagramConfig 增加并发配置

修改文件：

- src/main/java/com/ywt/passage/config/SvgDiagramConfig.java

建议新增字段：

```java
/**
 * SVG 同源并发上限
 */
private Integer maxConcurrency = 2;
```

如果你后面还会继续做缓存，这个类还会再扩字段，所以这次加完后不要把它写死成只服务并发改造的形状。

### 第 2 步：在 application.yaml 增加配置项

修改文件：

- src/main/resources/application.yaml

建议新增：

```yaml
svg-diagram:
  max-concurrency: 2
```

如果后面继续做缓存，可以把缓存相关配置放在同一个 svg-diagram 节点下，最终合并为一个完整配置块。

### 第 3 步：在 AsyncConfig 里增加 SVG 专用线程池

修改文件：

- src/main/java/com/ywt/passage/config/AsyncConfig.java

推荐新增一个 Bean，例如：

```java
@Bean(name = "svgDiagramExecutor")
public Executor svgDiagramExecutor(SvgDiagramConfig svgDiagramConfig) {
    int concurrency = Math.max(1, svgDiagramConfig.getMaxConcurrency());

    ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
    executor.setCorePoolSize(concurrency);
    executor.setMaxPoolSize(concurrency);
    executor.setQueueCapacity(concurrency * 4);
    executor.setThreadNamePrefix("svg-diagram-");
    executor.setRejectedExecutionHandler(new ThreadPoolExecutor.CallerRunsPolicy());
    executor.setWaitForTasksToCompleteOnShutdown(true);
    executor.setAwaitTerminationSeconds(60);
    executor.initialize();
    return executor;
}
```

这里建议单独加线程池，而不是复用 articleExecutor，原因有两个：

- articleExecutor 是文章阶段任务的外层线程池，不适合再承接大量 SVG 子任务
- 单独线程池更容易把“SVG 最大并发数”控制在一个稳定值上

### 第 4 步：在 ParallelImageGenerator 注入配置和专用 Executor

修改文件：

- src/main/java/com/ywt/passage/agent/parallel/ParallelImageGenerator.java

建议新增依赖：

- SvgDiagramConfig svgDiagramConfig
- Executor svgDiagramExecutor

如果你用的是构造器注入，可以把 svgDiagramExecutor 通过 @Qualifier("svgDiagramExecutor") 注进来。

例如：

```java
private final ImageGenerationTool imageGenerationTool;
private final SvgDiagramConfig svgDiagramConfig;
private final Executor svgDiagramExecutor;
```

如果构造器注入需要加 Qualifier，可以改为字段注入 svgDiagramExecutor，或者手写构造器。

## 5. ParallelImageGenerator 具体怎么改

这一步是整个并发改造的核心。

### 5.1 不要直接在现有 for 循环里硬塞 runAsync

当前 executeParallel 里的结构是：

- 每个来源 entry 启一个异步任务
- 异步任务内部 for req 串行执行

推荐改法不是直接把 for 循环替换成 runAsync，而是先把“单张图处理逻辑”抽成独立方法，再让 SVG 分支决定串行还是并行。

建议拆成这几个方法：

- executeParallel(...)
- processSourceGroup(...)
- processSingleRequirement(...)
- executeSvgRequirementsWithLimit(...)

### 5.2 推荐重构后的结构

建议把 executeParallel 改成这样的职责分工：

```java
private List<ArticleState.ImageResult> executeParallel(
        Map<String, List<ArticleState.ImageRequirement>> groupedBySource,
        Consumer<String> streamHandler) {

    CopyOnWriteArrayList<ArticleState.ImageResult> allImages = new CopyOnWriteArrayList<>();

    List<CompletableFuture<Void>> futures = groupedBySource.entrySet().stream()
            .map(entry -> CompletableFuture.runAsync(() -> {
                processSourceGroup(entry.getKey(), entry.getValue(), streamHandler, allImages);
            }))
            .toList();

    CompletableFuture.allOf(futures.toArray(new CompletableFuture[0])).join();
    return new ArrayList<>(allImages);
}
```

然后把来源组内逻辑单独抽出来：

```java
private void processSourceGroup(
        String imageSource,
        List<ArticleState.ImageRequirement> requirements,
        Consumer<String> streamHandler,
        CopyOnWriteArrayList<ArticleState.ImageResult> allImages) {

    ImageMethodEnum method = ImageMethodEnum.getByValue(imageSource);
    boolean useSvgLimitedConcurrency = method == ImageMethodEnum.SVG_DIAGRAM
            && requirements.size() > 1
            && svgDiagramConfig.getMaxConcurrency() != null
            && svgDiagramConfig.getMaxConcurrency() > 1;

    if (useSvgLimitedConcurrency) {
        executeSvgRequirementsWithLimit(requirements, streamHandler, allImages);
        return;
    }

    for (ArticleState.ImageRequirement req : requirements) {
        processSingleRequirement(req, streamHandler, allImages);
    }
}
```

### 5.3 单张图处理逻辑必须抽出来

你现在 for 循环里的这段逻辑要原样抽成 processSingleRequirement：

- 调 imageGenerationTool.generateImageDirect(...)
- 成功时 convertToImageResult
- allImages.add(...)
- 发 IMAGE_COMPLETE SSE
- 失败时写 warn/error 日志

建议方法签名：

```java
private void processSingleRequirement(
        ArticleState.ImageRequirement req,
        Consumer<String> streamHandler,
        CopyOnWriteArrayList<ArticleState.ImageResult> allImages)
```

这样做的好处是：

- 串行和并发都复用同一套单张图处理逻辑
- 不会出现“两份逻辑一份改了另一份忘了改”的问题

### 5.4 SVG 限流并发建议这么写

推荐新增一个专门方法：

```java
private void executeSvgRequirementsWithLimit(
        List<ArticleState.ImageRequirement> requirements,
        Consumer<String> streamHandler,
        CopyOnWriteArrayList<ArticleState.ImageResult> allImages) {

    List<CompletableFuture<Void>> futures = requirements.stream()
            .map(req -> CompletableFuture.runAsync(
                    () -> processSingleRequirement(req, streamHandler, allImages),
                    svgDiagramExecutor
            ))
            .toList();

    CompletableFuture.allOf(futures.toArray(new CompletableFuture[0])).join();
}
```

为什么这里不再额外加 Semaphore：

- 如果 svgDiagramExecutor 本身就是固定大小线程池，线程池容量已经天然形成并发上限
- 这比“公共线程池 + Semaphore”更直观，也更容易观察线程行为

### 5.5 结果顺序不要依赖执行顺序

并发后，IMAGE_COMPLETE 的到达顺序和 allImages 的加入顺序都会变成非确定的。

这不是 bug，但你必须保留 apply(...) 里最后这段排序：

```java
allImages.sort((a, b) -> {
    Integer posA = a.getPosition() != null ? a.getPosition() : 0;
    Integer posB = b.getPosition() != null ? b.getPosition() : 0;
    return posA.compareTo(posB);
});
```

这段不能删，也不能依赖并发结果自然有序。

## 6. 你实现时最容易踩的坑

### 6.1 不要把所有来源都改成同源并发

本次只建议放开 SVG_DIAGRAM。

原因：

- SVG 是纯生成型任务，收益最大
- 其他来源可能涉及外部 API、下载、限流、失败重试，行为不一定适合一起放开

### 6.2 不要丢掉 IMAGE_COMPLETE 的 SSE 推送

前端的配图进度条、日志、配图计数，都是靠 IMAGE_COMPLETE 在推。

如果你把处理逻辑拆分后忘了保留这段消息，前端会直接失去进度感知。

### 6.3 不要让 SVG 并发共用 articleExecutor

如果 articleExecutor 同时跑：

- 文章阶段外层异步任务
- SVG 内层并发子任务

后面排查线程占用会很乱，也不利于独立调参。

### 6.4 不要依赖 CopyOnWriteArrayList 的顺序

CopyOnWriteArrayList 线程安全没问题，但并发插入的先后顺序不可预测。最终排序仍然要以 position 为准。

### 6.5 maxConcurrency 要兜底成至少 1

配置错误时不要直接抛异常把整条图片链路打断。最稳妥的做法是：

- maxConcurrency <= 0 时，内部兜底成 1
- 也就是自动退回串行

## 7. 建议你手动实现时的改动顺序

建议按下面顺序手动改，不要一口气同时改完：

1. 先给 SvgDiagramConfig 和 application.yaml 加 max-concurrency
2. 再给 AsyncConfig 补 svgDiagramExecutor
3. 再重构 ParallelImageGenerator，把单张图处理逻辑抽成独立方法
4. 最后只给 SVG 分支接入 svgDiagramExecutor 并发执行
5. 编译通过后，再做一次真实流程验证

这样你每一步出问题时，回溯成本最低。

## 8. 验证方式

### 8.1 编译验证

先跑：

```bash
mvn -q -DskipTests compile
```

目标：

- 新增配置项后能正常绑定
- 新线程池 Bean 没有注入错误
- ParallelImageGenerator 的泛型和 CompletableFuture 没有编译错误

### 8.2 真实流程验证

固定一组输入跑完整流程：

- 主题：对日开发转国内开发
- 风格：情感风格
- 配图方式：仅 SVG

重点看这几个现象：

- 配图阶段总耗时是否明显下降
- 后端日志里是否能看到多张 SVG 几乎同时开始处理
- 前端 IMAGE_COMPLETE 计数是否仍然正常递增
- 最终图片顺序是否仍然按 position 正确落到正文中

### 8.3 观测日志建议

建议至少补这几类日志：

- 当前来源组开始处理：imageSource、数量
- SVG 分支是否启用限流并发：maxConcurrency、数量
- 单张图开始/成功/失败：position、imageSource
- 整个 SVG 组完成耗时

## 9. 预期结果

如果这次改造完成，理想结果应该是：

- 5 张 SVG 不再是纯串行
- 在 maxConcurrency=2 时，理论执行节奏变成“2 张并发 + 2 张并发 + 1 张收尾”
- 前端进度、日志、最终图文合成行为不变
- 你后面再做缓存时，不需要推翻这次改造
