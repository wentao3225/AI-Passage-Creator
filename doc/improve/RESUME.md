## 📝 简历项目描述（推荐版本）

### 版本 A：偏 AI 应用方向（如果是 AI 岗位）

```
AI-Passage-Creator（AI 长文创作系统）

基于 Spring Boot 3 + Vue 3 构建的人机协同 AI 内容创作平台，
支持用户输入选题后，通过多 Agent 编排自动完成标题生成、大纲生成、
正文生成、配图分析和图文合成，并提供 SSE 实时推送和并行配图优化。

核心职责：
- 设计并实现了 5 个 Agent（标题/大纲/正文/配图分析/图文合成），
  采用 StateGraph 模式串联执行流程，支持从任意阶段失败后重试
- 基于 SSE（SseEmitter）构建实时推送模块，支持大纲/正文流式输出
  和配图进度逐张推送，单任务推送 600-1200 条消息
- 实现多来源配图并行生成（Pexels、Mermaid、Iconify 等 5+ 来源），
  按来源分组并行执行，总时间从串行 11s 优化到 6s
- 设计三层容错机制（LLM JSON 解析容错 → 配图需求验证 → 执行降级），
  确保单点故障不影响整体流程
- 通过 AOP 自动采集每个 Agent 的执行日志（入参、出参、耗时），
  支持性能分析和故障回溯

技术栈：Spring Boot 3, MyBatis-Flex, SSE, DashScope, 
        Vue 3, Ant Design Vue, Pinia, Redis
```

### 版本 B：偏后端工程能力（如果是通用 Java 岗位）

```
AI-Passage-Creator（人机协同 AI 内容创作系统）

一个基于 Spring Boot 3 的异步任务编排系统，通过多 Agent 协作
完成从选题到成文的完整生成流程，支持流式推送、并行执行和容错恢复。

核心职责：
- 基于 Spring @Async + LangGraph StateGraph 实现异步任务编排，
  支持 5 阶段状态机驱动（PENDING → 标题 → 大纲 → 正文 → 完成）
- 使用 SseEmitter 构建实时消息推送模块，覆盖全流程 13 种消息类型
- 实现 CompletableFuture + imageSource 分组并行策略，
  同源串行、不同源并行，平衡性能和 API 限流约束
- 设计三层容错体系（LLM 输出解析 → 业务验证 → 自动降级）
- 通过 AOP 切面自动采集 Agent 执行日志（agent_log 表），
  用于性能分析和故障排查
- 前端使用 Vue 3 + Ant Design Vue + EventSource 实现流式展示

技术栈：Spring Boot 3, MyBatis-Flex, Redis, SSE, 
        Vue 3, Pinia, Ant Design Vue
```

## 🗣️ 3 分钟项目讲解话术

> 以下是你应该当面讲述的版本，控制在 3 分钟以内。

### 开场（15 秒）

"这个项目是一个 AI 长文创作系统。用户输入一个选题和风格，
系统会通过多 Agent 协作，自动完成标题生成、大纲生成、正文生成、
配图分析和图文合成——整个过程用户可以实时看到进度。"

### 核心问题（30 秒）

"现有方案的问题：一次性调用 LLM 生成，用户无法控制结果，
失败了要全部重来，中间过程是个黑盒。

我们的方案：拆成 5 个 Agent，每个 Agent 做一件事。
用户在标题选择和大纲编辑阶段可以确认和修改。
某个 Agent 失败，只需重跑该阶段，不丢之前的结果。"

### 技术亮点（1 分钟）

"技术上主要有三点：

第一，SSE 实时推送。用户能看到完整生成过程——大纲逐字出现、
正文流式输出、配图进度 1/10、2/10... 而不是提交后转圈。

第二，配图并行执行。5 个图片来源（Pexels、Mermaid、Iconify 等）
按来源分组并行，同源内部串行防限流，总时间从串行 11 秒降到 6 秒。

第三，多层容错。LLM 输出格式不稳定时自动修复；
某个配图来源失败时自动降级到另一个来源；
所有来源都失败时还有 PICSUM 兜底。没有单点故障。"

### 技术栈（30 秒）

"后端 Spring Boot 3 + MyBatis-Flex + Redis，
前端 Vue 3 + Ant Design Vue，
AI 调用通过阿里云 DashScope，
SSE 用 Spring 的 SseEmitter。"

### 我的角色（15 秒）

"从架构设计到代码实现都是我独立完成的。
包括 Agent 编排、SSE 推送、并行配图、容错、前端交互等。"

## 🔥 常见抗追问练习

### Q: 这个项目是你一个人做的？
**A**: "对，从需求分析到架构设计到编码实现都是我一个人完成的。
      这也是我把它作为主项目的原因——它能完整体现我的后端工程能力
      和 AI 应用开发能力。"

### Q: 跟市面上的 AI 写作工具比有什么不同？
**A**: "市面上的 AI 写作工具大多是端到端生成，用户只有'生成'和'重试'
      两个选项。我们的系统强调人机协同——用户在标题、大纲两阶段
      可以参与决策，能看到完整的生成过程，而且支持从任意阶段恢复。
      定位是'可控制的内容生产'而非'快速生成'。"

### Q: 生产环境有实际部署吗？性能怎么样？
**A**: "当前是个人项目，没有在生产环境部署。但设计时考虑了工程实践：
      并发控制、限流约束、缓存优化、日志采集等。如果有生产需求，
      配置 Nginx + 多实例部署即可。当前单实例支持数十人同时使用，
      瓶颈主要在 LLM 调用速度而非后端处理能力。"

### Q: 有监控和告警吗？
**A**: "当前没有接入 Prometheus + Grafana 这类专业监控。
      但内置了足够的可观测性——agent_log 表记录了每次 Agent 执行的
      耗时和状态，应用日志记录了关键事件。如果需要生产监控，
      接入 Spring Boot Actuator + Prometheus 可以快速补齐。"

### Q: 测试覆盖怎么样？
**A**: "当前以功能验证为主，没有完整的单元测试覆盖。
      这也是我后续计划补齐的部分。核心的 Agent 编排、
      SSE 推送和配图逻辑已经做了基本的功能验证，
      但单元测试和集成测试确实需要完善。"
