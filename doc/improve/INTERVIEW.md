## 1️⃣ 多 Agent 编排

### Q1: 为什么分成 5 个 Agent？不是调用一次 LLM 就生成完？

**回答框架**：
- 一次性生成的缺点：用户无控制权、幻觉风险高、失败需从头重来
- 分阶段的价值：用户在关键节点可确认/编辑，质量可控
- 工程化体现：每个 Agent 职责单一，可独立测试和优化

> "一次性生成看起来简单，但生产环境中有三个现实问题：
> 1. 用户对标题不满意怎么办？一次性生成无法调整。
> 2. 如果某一环节出错，需要全部重来。
> 3. 黑盒生成用户没有信任感，不知道进度。
>
> 所以我们拆成 5 个 Agent：标题、大纲、正文、配图分析、配图执行、图文合成。
> 每个 Agent 输入/输出清晰，用户在标题选择和大纲编辑阶段能参与决策。
> 某个 Agent 失败，只需从该阶段重试，不丢前面的结果。"

### Q2: Agent 编排怎么实现的？用的是什么框架？

**回答框架**：
- 基于 LangGraph 的 StateGraph 实现
- 每个 Agent 是一个 Node，Node 间通过 Edge 连接
- 状态管理通过 OverAllState 对象在 Node 间传递

> "我们用了类似 LangGraph 的 StateGraph 模式。每个 Agent 注册为一个 Node，
> 通过 Edge 定义执行顺序。核心状态通过 OverAllState 对象在 Node 之间传递。
>
> 比如 Phase3 的编排：
> - ContentGenerator → 生成正文
> - ImageAnalyzer → 分析配图需求（依赖正文完成）
> - ParallelImageGenerator → 并行生成配图
> - ContentMerger → 图文合成
>
> 这样新增 Agent 只需加 Node 和 Edge，不需要改 Controller 的逻辑。"

### Q3: StateGraph 和直接串行调用有什么区别？

**回答框架**：
- 串行调用：硬编码顺序，可读性差，扩展困难
- StateGraph：可视化流程、状态统一管理、易于扩展并行分支

> "串行调用就是写一个方法里按顺序调 A→B→C，如果需求变了改代码。
> StateGraph 的好处是：
> 1. 流程可视化——看一眼就知道执行顺序
> 2. 状态统一管理——Node 之间通过 State 传递数据，不依赖共享变量
> 3. 容易扩展——比如未来我想改为 正文生成 和 配图分析 异步并行，改 Edge 就行，
>    不用重写编排逻辑"

## 2️⃣ SSE 流式推送

### Q4: SSE 的实现细节是什么？为什么不用 WebSocket？

**回答框架**：
- 使用 Spring 的 SseEmitter
- 选 SSE 的理由：单向推送 + 原生自动重连
- 核心组件：SseEmitterManager + StreamHandlerContext（ThreadLocal）

> "后端用 Spring 的 SseEmitter，在前端用浏览器原生的 EventSource。
> 选 SSE 而不是 WebSocket 基于两个原因：
>
> 第一，我们的场景是后端→前端的单向推送，前端不需要反向控制 Agent 执行。
> 第二，手机网络不稳定时 SSE 浏览器的自动重连机制比 WebSocket 手动处理更可靠。
>
> 技术实现上，我们通过 ThreadLocal 存储流处理器，解决了 StateGraph 序列化
> 和流式输出之间的冲突——Agent 执行时直接从 ThreadLocal 拿到 SseEmitter 引用，
> 不需要把流处理器塞进 State 对象里。"

### Q5: 如何处理 SSE 连接断开？什么情况下会丢消息？

**回答框架**：
- 浏览器 3 秒自动重连
- 消息缺失时的业务处理（重试机制）
- 不会丢关键消息，因为已落库

> "SSE 断开后浏览器会自动按 3 秒间隔重连，这是原生支持的。
> 但确实可能丢消息——比如重连期间后端推送了一条 IMAGE_COMPLETE。
>
> 不过实际不需要担心，因为：
> 1. 所有最终结果都保存到了数据库（article 表）
> 2. 用户刷新页面时通过 GET /article/{taskId} 拿到最新完整数据
> 3. 如果关心实时进度，重新建立 SSE 连接即可——后端会继续推送后续消息
>
> 更严谨的方案是设计消息序列号 + 前端去重，但当前场景不需要这么重。"

### Q6: 推送了 1000 多条消息，对性能有影响吗？

**回答框架**：
- 每条消息很小（几个字节 ~ 几十 KB）
- SseEmitter 基于 HTTP Keep-Alive，连接复用
- 单连接 50-100 msg/s 完全没问题

> "600-1200 条消息看起来多，但每条消息很小：
> - AGENT_STREAMING 消息只包含 type + 文本块（几十字节）
> - IMAGE_COMPLETE 稍微大点（包含图片 URL 和位置，几 KB）
>
> SseEmitter 底层是 HTTP Keep-Alive 连接，复用 TCP 连接，不会频繁建连。
> 实测单连接推送 1200 条消息，内存占用不超过 1MB，CPU 忽略不计。
> 瓶颈不在 SSE 推送本身，而在 LLM 生成速度——这个比推送慢两个数量级。"

## 3️⃣ 并行配图与多源兜底

### Q7: 如果配图来源 A 失败，系统怎么处理？

**回答框架**：
- 降级链：原来源 → 第一备选 → 第二备选 → PICSUM 兜底
- 降级过程不影响其他来源
- 失败的请求被日志记录，不阻塞流程

> "有三级降级机制。
>
> 第一级：需求级降级。比如用户要求用 DALL-E，但 DALL-E API 返回 403，
> ImageAnalyzerAgent 会调用 buildFallbackPreference() 尝试 PEXELS → EMOJI_PACK → ...
>
> 第二级：执行级降级。如果 PEXELS 也失败，ImageServiceStrategy 会
> 自动调用下一候选来源，直到成功或候选列表用完。
>
> 第三级：全局兜底。如果所有候选都失败，最终返回一张 PICSUM 随机占位图，
> 保证文章至少有一张图，而不是生成失败。
>
> 而且注意，某个来源失败不影响其他来源——ParallelImageGenerator 中每个
> CompletableFuture 独立执行，异常只捕获不抛，单点故障不级联。"

### Q8: 为什么要支持那么多配图来源？一个不行吗？

**回答框架**：
- 每个来源专业不同：流程图（Mermaid）、图标（Iconify）、实拍（Pexels）
- 场景化决策：按位置/类型自动匹配最优来源
- 高可用：无单点故障

> "如果只用 PEXELS，会有三个问题：
> 1. 无法生成流程图、原理图——Pexels 是实拍图库
> 2. 找不到合适的图标——需要 Iconify 这种专业图标源
> 3. Pexels API 万一故障，整篇文章无图
>
> 所以我们设计的是场景化匹配：
> - 流程/结构类内容 → 优先 SVG_DIAGRAM 或 MERMAID（生成图+流程图）
> - 通用配图 → PEXELS 或 EMOJI_PACK（快速返回，保证速度）
> - 小图标 → ICONIFY（专业图标库）
>
> 这样既保证了图片质量，也提升了系统可用性。
> 某个来源故障，自动降级到其他来源，用户无感知。"

### Q9: 并行执行时怎么控制并发度？不怕 API 限流？

**回答框架**：
- 同一来源内部串行执行
- SVG 来源额外限流（maxConcurrency=1-2）
- 不同来源之间并行

> "我们特别注意了限流问题。做法是：
>
> 1. 按 imageSource 分组——所有 PEXELS 的图片放在一个组
> 2. 同一组内部串行执行——不突发请求 PEXELS
> 3. 不同组之间并行——PEXELS 和 MERMAID 互不影响
> 4. SVG 来源额外加了个信号量限制——maxConcurrency=1 或 2，防止同时
>    调用多次 LLM 产生大量成本
>
> 这样总时间 = max(PEXELS 总耗时, MERMAID 总耗时, SVG 总耗时)，
> 而不是所有来源串行求和。比如 PEXELS 有 3 张图各 2 秒 = 串行 6 秒，
> 同时 SVG 有 1 张图 3 秒 = 3 秒，总时间 = max(6, 3) = 6 秒。
> 如果全串行 = 6 + 3 = 9 秒。"

## 4️⃣ 人机协同与交互

### Q10: 用户刷新页面后，系统怎么恢复到之前的状态？

**回答框架**：
- 核心：Article.phase 字段是"唯一真源"
- 前端加载时根据 phase 恢复到对应 UI 阶段
- 如果正在生成中，重新建立 SSE 连接

> "状态恢复的核心是数据库里的 phase 字段。
>
> 用户刷新页面后，前端会调用 GET /article/{taskId} 获取最新数据。
> 后端返回完整的 ArticleVO，其中包含 phase 字段。
>
> 前端拿到 phase 后：
> - phase=TITLE_SELECTING → 显示标题选择面板
> - phase=OUTLINE_EDITING → 从 article.outline 恢复大纲编辑器
> - phase=CONTENT_GENERATING → 重新建立 SSE 连接，继续接收推送
> - phase=COMPLETED → 直接展示最终文章
>
> 关键设计：不依赖于前端内存状态，所有关键数据已持久化。"

### Q11: 用户点"重试"时，前端和后端分别做了什么？

**回答框架**：
- 前端：重新建立 SSE 连接 + 重置本地状态 + 显示"生成中"
- 后端：清空该阶段结果 + 重新触发 @Async + 推送新 SSE 流

> "前端做三件事：
> 1. 断开旧的 SSE 连接
> 2. 重置本地状态（清空当前阶段的内容、进度条归零）
> 3. 调用 POST /restart-phase 发起重试
>
> 后端：
> 1. 检查 phase 是否合法（比如不能在 COMPLETED 时重试）
> 2. 清空相关字段（重试大纲生成就清空 outline）
> 3. 更新 phase 为生成中（OUTLINE_GENERATING）
> 4. 重新调用 @Async 方法
> 5. 新的 SSE 流重新推送
>
> 用户体验上，重试后用户看到的流程和首次一样，不会丢失上下文。"

## 5️⃣ 日志追踪与可观测性

### Q12: agent_log 表记录了什么？怎么用？

**回答框架**：
- AOP 自动拦截 @AgentExecution 注解
- 记录：入参、出参、耗时、状态、异常信息
- 用途：性能分析、故障排查、执行审计

> "agent_log 通过 AOP 切面自动记录。我们在每个 Agent 的方法上加了
> @AgentExecution 注解，切面会在这个方法执行前后自动采集数据。
>
> 每一条日志包含：
> - agent_name（哪个 Agent）
> - task_id（哪篇文章）
> - duration_ms（执行耗时，毫秒）
> - status（SUCCESS/FAILED）
> - error_msg（失败时的错误信息）
> - input_data / output_data（入参出参的 JSON 快照）
>
> 实际使用场景：
> 1. 性能分析：查询各 Agent 平均耗时，定位慢节点
> 2. 故障排查：某个任务失败，查 agent_log 看哪个 Agent 出错
> 3. 成本统计：统计 Agent 调用次数，估算 LLM 成本"

### Q13: 项目怎么保证"能抗追问"？

> "有三个原则：
>
> 1. 每个决策都有取舍。比如分阶段牺牲了速度但提升了可控性，
>    这种取舍能讲清楚，而不是说什么都是好。
>
> 2. 每个实现都基于真实场景。比如配图缓存只做了 SVG，因为 Pexels
>    搜索结果的缓存价值不大、LLM 生成 SVG 成本高所以缓存 ROI 高。
>
> 3. 知道哪里做得不够好。比如当前没有加乐观锁，没有做消息序列号，
>    如果面试官问我会说'这是预留的优化方向，当前场景下不是瓶颈'。"
