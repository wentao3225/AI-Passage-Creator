# AI-Passage-Creator SVG 本地缓存改造实施文档

## 1. 目标

本文只讲一件事：在不改动现有图片服务接口形状的前提下，为 SvgDiagramService 增加“本地文件缓存”，避免相同或高相似 SVG 需求反复调用大模型。

这次缓存改造的目标是：

- 缓存的是“已验证通过的 SVG 结果”
- 命中缓存时不再调用 chatModel
- 仍然走现有 ImageSearchService -> ImageServiceStrategy -> LocalImageStorageService 这条主链路
- 即使缓存失效或缓存异常，也不能影响主流程继续生成

## 2. 当前代码锚点

这次缓存改造重点看 5 个文件：

- src/main/java/com/ywt/passage/core/ImageSearch/SvgDiagramService.java
- src/main/java/com/ywt/passage/config/SvgDiagramConfig.java
- src/main/java/com/ywt/passage/model/dto/image/ImageData.java
- src/main/java/com/ywt/passage/service/ImageServiceStrategy.java
- src/main/resources/application.yaml

当前现状：

### 2.1 SvgDiagramService 每次都直调 LLM

当前 generateSvgDiagramData(requirement) 的流程是：

1. 校验 requirement 非空
2. callLlmToGenerateSvg(requirement)
3. 校验 SVG 格式
4. 转 byte[]
5. 返回 ImageData.fromBytes(...)

它没有任何缓存命中逻辑。

### 2.2 ImageServiceStrategy 已经支持直接接收字节数据

这点很关键。

当前 ImageServiceStrategy.getImageAndStore(...) 会拿到 ImageData，然后交给 LocalImageStorageService.uploadImageData(...)。

而 LocalImageStorageService 已经支持：

- URL
- DATA_URL
- BYTES

这意味着缓存命中时，只要 SvgDiagramService 返回 ImageData.fromBytes(...)，后面整条链路都不需要改。

### 2.3 当前最适合先做“缓存 SVG 内容”，而不是缓存最终 URL

原因：

- 当前服务接口返回的是 ImageData，不是“缓存实体”
- 如果缓存最终 URL，会涉及文件生命周期、静态资源可用性、URL 稳定性等额外问题
- 如果缓存原始 SVG XML 或 SVG 字节，SvgDiagramService 命中缓存后直接返回 ImageData.fromBytes(...) 就够了

所以第一阶段最推荐的做法是：

- 缓存“SVG 源文本”或“SVG 字节”
- 让 ImageServiceStrategy 继续负责每次落盘并返回访问 URL

这样实现最稳，改动也最小。

## 3. 推荐实现策略

推荐实现为：本地文件缓存 + TTL + 按规范化 requirement 计算 hash key。

不推荐一上来做：

- Redis 缓存
- 数据库存储缓存表
- 缓存最终对外 URL

推荐理由：

- 当前项目里还没有现成的 SVG 缓存设施
- 本地文件缓存最容易落地，调试成本最低
- 你这次是自己手动实现，本地文件缓存最适合作为第一版

## 4. 改动总览

建议这次缓存改造按下面顺序做：

### 第 1 步：给 SvgDiagramConfig 增加缓存字段

修改文件：

- src/main/java/com/ywt/passage/config/SvgDiagramConfig.java

建议新增字段：

```java
/** 是否启用 SVG 缓存 */
private Boolean cacheEnabled = true;

/** SVG 缓存 TTL（分钟） */
private Long cacheTtlMinutes = 1440L;

/** SVG 缓存目录 */
private String cacheDir = "tmp/svg-diagram-cache";
```

如果你已经按并发文档实现过 maxConcurrency，这个类最终会同时包含：

- maxConcurrency
- cacheEnabled
- cacheTtlMinutes
- cacheDir

所以改的时候要做字段合并，不要相互覆盖。

### 第 2 步：在 application.yaml 增加缓存配置

修改文件：

- src/main/resources/application.yaml

建议新增或合并：

```yaml
svg-diagram:
  max-concurrency: 2
  cache-enabled: true
  cache-ttl-minutes: 1440
  cache-dir: tmp/svg-diagram-cache
```

如果你还没做并发，也可以先只加缓存相关字段。

### 第 3 步：新增 SvgDiagramCacheManager 独立类

修改文件：

- src/main/java/com/ywt/passage/core/ImageSearch/SvgDiagramCacheManager.java

建议把所有缓存相关的方法都放进这个类，不要继续塞进 SvgDiagramService。这样可以避免：

- SvgDiagramService 同时承担“LLM 编排 + 缓存 IO + TTL 判断 + 文件清理”四类职责
- 后续缓存策略继续演进时，Service 变成大泥球
- 读写缓存逻辑分散在多个私有方法里，不利于调试和复用

建议放进去的方法包括：

- normalizeRequirement(...)
- tryReadCache(...)
- writeCache(...)
- buildCacheKey(...)
- resolveCacheRoot()
- resolveCacheFile(...)
- isCacheExpired(...)
- deleteQuietly(...)
- isSvgContentValid(...)

### 第 4 步：让 SvgDiagramService 只负责编排主流程

修改文件：

- src/main/java/com/ywt/passage/core/ImageSearch/SvgDiagramService.java

推荐把 generateSvgDiagramData(requirement) 的逻辑改成：

1. 校验 requirement 非空
2. 调用 SvgDiagramCacheManager.normalizeRequirement(...) 规范化 requirement
3. 调用 SvgDiagramCacheManager.tryReadCache(...) 先尝试读缓存
4. 命中缓存则直接返回 ImageData.fromBytes(...)
5. 未命中缓存再调 LLM
6. 生成成功且校验通过后，调用 SvgDiagramCacheManager.writeCache(...)
7. 最后返回 ImageData.fromBytes(...)

## 5. 建议新增 SvgDiagramCacheManager 独立类

### 5.1 为什么要单独抽类

如果继续把缓存方法都写在 SvgDiagramService 里，短期看省文件，长期一定会乱。更稳的做法是：

- SvgDiagramService 只负责“调 LLM、做流程编排、返回 ImageData”
- SvgDiagramCacheManager 专门负责“缓存 key、文件路径、TTL、读写、清理”

这样拆完之后，主流程会非常清楚，缓存策略也更容易单独维护。

### 5.2 新类建议放在哪个文件

建议新增：

- src/main/java/com/ywt/passage/core/ImageSearch/SvgDiagramCacheManager.java

第一版建议直接和 SvgDiagramService 放在同一个 package 下：

- 不需要额外调整包扫描
- import 最简单
- 你后面如果要再拆成 cache 子包，也很容易迁移

### 5.3 这个类里建议包含哪些方法

建议这个类里直接包含下面这些方法：

- public String normalizeRequirement(String requirement)
- public ImageData tryReadCache(String normalizedRequirement)
- public void writeCache(String normalizedRequirement, String svgCode)
- private String buildCacheKey(String normalizedRequirement)
- private Path resolveCacheRoot()
- private Path resolveCacheFile(String normalizedRequirement)
- private boolean isCacheExpired(Path cacheFile)
- private void deleteQuietly(Path file)
- private boolean isSvgContentValid(String svgCode)

这样缓存读写相关的方法就不会再散落在 SvgDiagramService 里。

### 5.4 SvgDiagramCacheManager 完整版代码

建议新增：

```java
package com.ywt.passage.core.ImageSearch;

import cn.hutool.core.util.StrUtil;
import cn.hutool.crypto.SecureUtil;
import com.ywt.passage.config.SvgDiagramConfig;
import com.ywt.passage.model.dto.image.ImageData;
import jakarta.annotation.Resource;
import lombok.extern.slf4j.Slf4j;
import org.springframework.stereotype.Component;

import java.io.IOException;
import java.nio.charset.StandardCharsets;
import java.nio.file.AtomicMoveNotSupportedException;
import java.nio.file.Files;
import java.nio.file.Path;
import java.nio.file.Paths;
import java.nio.file.StandardCopyOption;
import java.nio.file.StandardOpenOption;
import java.nio.file.attribute.FileTime;
import java.time.Duration;
import java.time.Instant;

/**
 * SVG 本地缓存管理器
 * 负责 requirement 规范化、缓存 key 计算、缓存文件读写、TTL 判断和坏缓存清理
 */
@Component
@Slf4j
public class SvgDiagramCacheManager {

    private static final String SVG_CONTENT_TYPE = "image/svg+xml";
    private static final String CACHE_FILE_SUFFIX = ".svg";
    private static final String TEMP_FILE_SUFFIX = ".tmp";

    @Resource
    private SvgDiagramConfig svgDiagramConfig;

    /**
     * 规范化 requirement
     * 只做 trim 和连续空白折叠，避免过度清洗导致错误归并
     */
    public String normalizeRequirement(String requirement) {
        if (StrUtil.isBlank(requirement)) {
            return "";
        }
        return requirement.trim().replaceAll("\\s+", " ");
    }

    /**
     * 尝试读取 SVG 本地缓存
     */
    public ImageData tryReadCache(String normalizedRequirement) {
        if (Boolean.FALSE.equals(svgDiagramConfig.getCacheEnabled())) {
            return null;
        }
        if (StrUtil.isBlank(normalizedRequirement)) {
            return null;
        }

        String cacheKey = buildCacheKey(normalizedRequirement);
        Path cacheFile = resolveCacheFile(normalizedRequirement);

        try {
            if (!Files.exists(cacheFile)) {
                log.info("SVG 缓存未命中, key={}", cacheKey);
                return null;
            }

            if (isCacheExpired(cacheFile)) {
                log.info("SVG 缓存已过期, key={}, file={}", cacheKey, cacheFile);
                deleteQuietly(cacheFile);
                return null;
            }

            String svgCode = Files.readString(cacheFile, StandardCharsets.UTF_8);
            if (!isSvgContentValid(svgCode)) {
                log.warn("SVG 缓存内容无效, key={}, file={}", cacheKey, cacheFile);
                deleteQuietly(cacheFile);
                return null;
            }

            byte[] svgBytes = svgCode.getBytes(StandardCharsets.UTF_8);
            log.info("SVG 缓存命中, key={}, file={}, size={} bytes",
                    cacheKey, cacheFile, svgBytes.length);
            return ImageData.fromBytes(svgBytes, SVG_CONTENT_TYPE);
        } catch (Exception e) {
            log.warn("SVG 缓存读取失败, key={}", cacheKey, e);
            return null;
        }
    }

    /**
     * 写入 SVG 本地缓存
     */
    public void writeCache(String normalizedRequirement, String svgCode) {
        if (Boolean.FALSE.equals(svgDiagramConfig.getCacheEnabled())) {
            return;
        }
        if (StrUtil.isBlank(normalizedRequirement) || StrUtil.isBlank(svgCode)) {
            return;
        }
        if (!isSvgContentValid(svgCode)) {
            log.warn("跳过写入 SVG 缓存，内容校验未通过");
            return;
        }

        String cacheKey = buildCacheKey(normalizedRequirement);
        Path targetFile = resolveCacheFile(normalizedRequirement);
        Path cacheRoot = targetFile.getParent();
        Path tempFile = cacheRoot.resolve(cacheKey + TEMP_FILE_SUFFIX);

        try {
            Files.createDirectories(cacheRoot);
            Files.writeString(tempFile, svgCode, StandardCharsets.UTF_8,
                    StandardOpenOption.CREATE,
                    StandardOpenOption.TRUNCATE_EXISTING,
                    StandardOpenOption.WRITE);

            try {
                Files.move(tempFile, targetFile,
                        StandardCopyOption.REPLACE_EXISTING,
                        StandardCopyOption.ATOMIC_MOVE);
            } catch (AtomicMoveNotSupportedException e) {
                Files.move(tempFile, targetFile, StandardCopyOption.REPLACE_EXISTING);
            }

            log.info("SVG 缓存写入成功, key={}, file={}", cacheKey, targetFile);
        } catch (Exception e) {
            log.warn("SVG 缓存写入失败, key={}", cacheKey, e);
            deleteQuietly(tempFile);
        }
    }

    /**
     * 基于规范化 requirement 构造缓存 key
     */
    private String buildCacheKey(String normalizedRequirement) {
        return SecureUtil.md5(normalizedRequirement);
    }

    /**
     * 解析缓存根目录
     */
    private Path resolveCacheRoot() {
        String cacheDir = svgDiagramConfig.getCacheDir();
        if (StrUtil.isBlank(cacheDir)) {
            cacheDir = "tmp/svg-diagram-cache";
        }
        return Paths.get(cacheDir).toAbsolutePath().normalize();
    }

    /**
     * 解析某个 requirement 对应的缓存文件路径
     */
    private Path resolveCacheFile(String normalizedRequirement) {
        String cacheKey = buildCacheKey(normalizedRequirement);
        return resolveCacheRoot().resolve(cacheKey + CACHE_FILE_SUFFIX);
    }

    /**
     * 判断缓存是否过期
     */
    private boolean isCacheExpired(Path cacheFile) throws IOException {
        Long cacheTtlMinutes = svgDiagramConfig.getCacheTtlMinutes();
        if (cacheTtlMinutes == null || cacheTtlMinutes <= 0) {
            return false;
        }

        FileTime lastModifiedTime = Files.getLastModifiedTime(cacheFile);
        Instant expireAt = lastModifiedTime.toInstant().plus(Duration.ofMinutes(cacheTtlMinutes));
        return Instant.now().isAfter(expireAt);
    }

    /**
     * 安静删除缓存文件，失败时只记日志
     */
    private void deleteQuietly(Path file) {
        if (file == null) {
            return;
        }

        try {
            Files.deleteIfExists(file);
        } catch (Exception e) {
            log.warn("删除 SVG 缓存文件失败, file={}", file, e);
        }
    }

    /**
     * 校验缓存中的 SVG 内容是否满足最基本格式要求
     */
    private boolean isSvgContentValid(String svgCode) {
        return StrUtil.isNotBlank(svgCode)
                && svgCode.contains("<svg")
                && svgCode.contains("</svg>");
    }
}
```

这个类已经把缓存相关的方法都收口到一起了。你如果按这个版本落地，缓存逻辑就不会继续污染 SvgDiagramService。

### 5.5 SvgDiagramService 如何接入这个新类

改完之后，SvgDiagramService 顶层流程建议改成下面这样。这里的重点不是再写一堆缓存私有方法，而是只保留流程编排。

```java
@Service
@Slf4j
public class SvgDiagramService implements ImageSearchService {

    @Resource
    private DashScopeChatModel chatModel;

    @Resource
    private SvgDiagramCacheManager svgDiagramCacheManager;

    @Override
    public ImageData getImageData(ImageRequest request) {
        String requirement = request.getEffectiveParam(true);
        return generateSvgDiagramData(requirement);
    }

    public ImageData generateSvgDiagramData(String requirement) {
        if (StrUtil.isBlank(requirement)) {
            log.warn("SVG 图表需求为空");
            return null;
        }

        long start = System.currentTimeMillis();
        String normalizedRequirement = svgDiagramCacheManager.normalizeRequirement(requirement);

        try {
            ImageData cached = svgDiagramCacheManager.tryReadCache(normalizedRequirement);
            if (cached != null) {
                log.info("SVG 缓存命中, cost={}ms", System.currentTimeMillis() - start);
                return cached;
            }

            String svgCode = callLlmToGenerateSvg(normalizedRequirement);
            if (StrUtil.isBlank(svgCode)) {
                log.error("LLM 未生成 SVG 代码");
                return null;
            }

            if (!isValidSvg(svgCode)) {
                log.error("生成的 SVG 代码格式无效");
                return null;
            }

            svgDiagramCacheManager.writeCache(normalizedRequirement, svgCode);

            byte[] svgBytes = svgCode.getBytes(StandardCharsets.UTF_8);
            log.info("SVG 概念示意图生成成功, size={} bytes, cost={}ms",
                    svgBytes.length, System.currentTimeMillis() - start);
            return ImageData.fromBytes(svgBytes, "image/svg+xml");
        } catch (Exception e) {
            log.error("SVG 概念示意图生成异常, requirement={}", requirement, e);
            return null;
        }
    }

    private String callLlmToGenerateSvg(String requirement) {
        // 保持你当前实现不变
    }

    private boolean isValidSvg(String svgCode) {
        // 保持你当前实现不变
    }
}
```

这个写法的重点是：

- Service 不再关心 cacheDir、TTL、临时文件、坏缓存删除这些细节
- cacheEnabled 的开关判断也统一收口到 SvgDiagramCacheManager 内部
- 主流程只剩“查缓存 -> 调 LLM -> 校验 -> 写缓存 -> 返回结果”

## 6. 每个缓存方法的职责说明

### 6.1 normalizeRequirement

职责：

- 统一做 requirement 的轻量规范化
- 保证相同语义、仅空白差异的输入能落到同一个缓存 key
- 不做过度清洗，避免不同需求被误判成同一个缓存项

### 6.2 tryReadCache

职责：

- 尝试命中本地缓存
- 自动处理 miss、expired、invalid cache 三种场景
- 命中后直接返回 ImageData.fromBytes(...)
- 任何异常都降级成返回 null，不影响主流程继续调 LLM

### 6.3 writeCache

职责：

- 在 SVG 内容校验通过后再落缓存
- 通过临时文件 + move 降低半截文件风险
- 写入失败只记日志，不中断当前请求

### 6.4 buildCacheKey

职责：

- 基于 normalizedRequirement 生成稳定 hash
- 隔离“key 算法”这个变化点，后续改 MD5、SHA-256 都只改一个方法

### 6.5 resolveCacheRoot / resolveCacheFile

职责：

- 把缓存目录解析逻辑和缓存文件路径拼装逻辑封装起来
- 避免在读写缓存的主流程里到处拼字符串路径

### 6.6 isCacheExpired

职责：

- 根据文件最后修改时间和 cacheTtlMinutes 判断是否过期
- TTL <= 0 时默认视为永不过期，避免误删

### 6.7 deleteQuietly

职责：

- 在删过期缓存、删坏缓存、删临时文件失败时吞掉异常
- 保证缓存清理问题不会拖垮主链路

### 6.8 isSvgContentValid

职责：

- 对缓存读写阶段做一层最基本的 SVG 合法性兜底
- 第一版先用简单规则落地，后续如果你把 SVG 校验抽成共享 Validator，再统一收口过去

## 7. 你实现时最容易踩的坑

### 7.1 不要缓存未校验的 SVG

一定要在 isValidSvg(svgCode) 通过之后再写缓存。

否则一旦缓存里写进去坏 SVG，后面每次命中都会直接复用坏结果。

### 7.2 不要缓存最终访问 URL

当前第一版缓存不要缓存：

- upload 后的最终 URL
- 静态资源访问地址

因为这样会把缓存生命周期和上传文件生命周期强行耦合在一起。

第一版最稳的方式是：

- 缓存 SVG 内容
- 命中后重新返回 ImageData.fromBytes(...)
- 让 LocalImageStorageService 继续负责最终落盘

### 7.3 命中缓存后仍会生成一个新的上传文件，这是允许的

这是第一版最容易被误解的点。

缓存命中时：

- 你确实绕过了 LLM
- 但 LocalImageStorageService 仍会把这份 SVG 字节再落一次盘

这会带来重复文件，但这不是第一阶段的核心问题。

第一阶段只解决：

- 大模型耗时
- 重复生成耗时

不要为了“复用同一个最终 URL”把改动面扩大到整个图片存储链路。

### 7.4 缓存异常不能影响主流程

任何缓存相关失败都必须退化成“当次不走缓存、直接调 LLM”。

不能因为：

- 路径不存在
- 文件读写失败
- 计算 key 异常
- 缓存文件坏掉

就把整条 SVG 生成链路打断。

### 7.5 cacheDir 不要放到 static/uploads 下面

缓存目录建议和最终对外访问目录分开。

不要把缓存直接放进：

- src/main/resources/static/uploads

因为缓存不是对外资源，不需要暴露成静态访问路径。

## 8. 建议你手动实现时的顺序

建议按这个顺序手动改：

1. 先给 SvgDiagramConfig 和 application.yaml 加缓存字段
2. 新增 SvgDiagramCacheManager，并把缓存相关的方法全部写进去
3. 在 SvgDiagramService 里注入 SvgDiagramCacheManager
4. 再改 generateSvgDiagramData 主流程，只保留编排逻辑
5. 编译通过后，再用重复输入做实际命中验证

这样比一口气全改更稳。

## 10. 预期结果

如果这次缓存改造完成，理想结果应该是：

- 相同或高度相似的 SVG 需求不会重复调用大模型
- 第二次及后续同类任务的 SVG 阶段耗时明显下降
- 缓存异常不会拖垮主链路
- 这次缓存改造不会阻塞你后面继续做模板快路径或结构化渲染

- src/main/java/com/ywt/passage/config/SvgDiagramConfig.java
- src/main/java/com/ywt/passage/core/ImageSearch/SvgDiagramCacheManager.java
- src/main/java/com/ywt/passage/core/ImageSearch/SvgDiagramService.java

- src/main/java/com/ywt/passage/config/AsyncConfig.java
- src/main/java/com/ywt/passage/agent/parallel/ParallelImageGenerator.java

但缓存第一版本身不要求你改这两个文件。
