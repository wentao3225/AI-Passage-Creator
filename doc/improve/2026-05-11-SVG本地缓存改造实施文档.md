# AI-Passage-Creator SVG 本地缓存改造实施文档

## 1. 目标

本文只讲一件事：在不改动现有图片服务接口形状的前提下，为 SvgDiagramService 增加“本地文件缓存”，避免相同或高相似 SVG 需求反复调用大模型。

这次缓存改造的目标是：

- 缓存的是“已验证通过的 SVG 结果”
- 命中缓存时不再调用 chatModel
- 仍然走现有 ImageSearchService -> ImageServiceStrategy -> LocalImageStorageService 这条主链路
- 即使缓存失效或缓存异常，也不能影响主流程继续生成

## 2. 当前代码锚点

这次缓存改造重点看 5 个文件：

- src/main/java/com/ywt/passage/core/ImageSearch/SvgDiagramService.java
- src/main/java/com/ywt/passage/config/SvgDiagramConfig.java
- src/main/java/com/ywt/passage/model/dto/image/ImageData.java
- src/main/java/com/ywt/passage/service/ImageServiceStrategy.java
- src/main/resources/application.yaml

当前现状：

### 2.1 SvgDiagramService 每次都直调 LLM

当前 generateSvgDiagramData(requirement) 的流程是：

1. 校验 requirement 非空
2. callLlmToGenerateSvg(requirement)
3. 校验 SVG 格式
4. 转 byte[]
5. 返回 ImageData.fromBytes(...)

它没有任何缓存命中逻辑。

### 2.2 ImageServiceStrategy 已经支持直接接收字节数据

这点很关键。

当前 ImageServiceStrategy.getImageAndStore(...) 会拿到 ImageData，然后交给 LocalImageStorageService.uploadImageData(...)。

而 LocalImageStorageService 已经支持：

- URL
- DATA_URL
- BYTES

这意味着缓存命中时，只要 SvgDiagramService 返回 ImageData.fromBytes(...)，后面整条链路都不需要改。

### 2.3 当前最适合先做“缓存 SVG 内容”，而不是缓存最终 URL

原因：

- 当前服务接口返回的是 ImageData，不是“缓存实体”
- 如果缓存最终 URL，会涉及文件生命周期、静态资源可用性、URL 稳定性等额外问题
- 如果缓存原始 SVG XML 或 SVG 字节，SvgDiagramService 命中缓存后直接返回 ImageData.fromBytes(...) 就够了

所以第一阶段最推荐的做法是：

- 缓存“SVG 源文本”或“SVG 字节”
- 让 ImageServiceStrategy 继续负责每次落盘并返回访问 URL

这样实现最稳，改动也最小。

## 3. 推荐实现策略

推荐实现为：本地文件缓存 + TTL + 按规范化 requirement 计算 hash key。

不推荐一上来做：

- Redis 缓存
- 数据库存储缓存表
- 缓存最终对外 URL

推荐理由：

- 当前项目里还没有现成的 SVG 缓存设施
- 本地文件缓存最容易落地，调试成本最低
- 你这次是自己手动实现，本地文件缓存最适合作为第一版

## 4. 改动总览

建议这次缓存改造按下面顺序做：

### 第 1 步：给 SvgDiagramConfig 增加缓存字段

修改文件：

- src/main/java/com/ywt/passage/config/SvgDiagramConfig.java

建议新增字段：

```java
/** 是否启用 SVG 缓存 */
private Boolean cacheEnabled = true;

/** SVG 缓存 TTL（分钟） */
private Long cacheTtlMinutes = 1440L;

/** SVG 缓存目录 */
private String cacheDir = "tmp/svg-diagram-cache";
```

如果你已经按并发文档实现过 maxConcurrency，这个类最终会同时包含：

- maxConcurrency
- cacheEnabled
- cacheTtlMinutes
- cacheDir

所以改的时候要做字段合并，不要相互覆盖。

### 第 2 步：在 application.yaml 增加缓存配置

修改文件：

- src/main/resources/application.yaml

建议新增或合并：

```yaml
svg-diagram:
  max-concurrency: 2
  cache-enabled: true
  cache-ttl-minutes: 1440
  cache-dir: tmp/svg-diagram-cache
```

如果你还没做并发，也可以先只加缓存相关字段。

### 第 3 步：在 SvgDiagramService 顶层流程里接入“先查缓存，再调 LLM”

修改文件：

- src/main/java/com/ywt/passage/core/ImageSearch/SvgDiagramService.java

推荐把 generateSvgDiagramData(requirement) 的逻辑改成：

1. 校验 requirement 非空
2. 规范化 requirement
3. 如果 cacheEnabled = true，先尝试读缓存
4. 命中缓存则直接返回 ImageData.fromBytes(...)
5. 未命中缓存再调 LLM
6. 生成成功且校验通过后写缓存
7. 最后返回 ImageData.fromBytes(...)

## 5. SvgDiagramService 具体怎么改

### 5.1 建议先补 3 个辅助方法

推荐先补下面 3 组辅助方法，再改主流程：

- normalizeRequirement(...)
- buildCacheKey(...)
- tryReadCache(...) / writeCache(...)

这样主流程会更清楚。

### 5.2 requirement 规范化不要太激进

建议新增：

```java
private String normalizeRequirement(String requirement) {
    if (StrUtil.isBlank(requirement)) {
        return "";
    }
    return requirement.trim().replaceAll("\\s+", " ");
}
```

这里建议只做：

- trim
- 连续空白折叠

不要做：

- 全量转小写
- 删除标点
- 过度清洗中文词序

原因是：过强规范化容易把“本来不同的需求”错误归并到同一个缓存 key。

### 5.3 cache key 建议用 MD5 或 SHA-256

你可以新增：

```java
private String buildCacheKey(String normalizedRequirement) {
    // 这里返回 32 位 md5 或 64 位 sha256 都可以
}
```

实现方式任选一种：

- java.security.MessageDigest
- Hutool DigestUtil

如果项目里没有现成 digest 工具，直接用 JDK 的 MessageDigest 就够了。

建议 key 只基于 normalizedRequirement，不要把时间、position 这类不影响 SVG 结构的字段混进来。

### 5.4 缓存文件建议直接存 .svg 文本

缓存目录结构建议简单一点：

```text
{cacheDir}/
  a1b2c3d4e5f6....svg
```

第一版不需要多级目录、不需要数据库索引、不需要元数据表。

TTL 可以直接依赖文件最后修改时间判断，不需要单独 sidecar 文件。

### 5.5 推荐主流程改法

建议把 generateSvgDiagramData(...) 改成接近下面这个结构：

```java
public ImageData generateSvgDiagramData(String requirement) {
    if (StrUtil.isBlank(requirement)) {
        log.warn("SVG 图表需求为空");
        return null;
    }

    long start = System.currentTimeMillis();
    String normalizedRequirement = normalizeRequirement(requirement);

    try {
        if (Boolean.TRUE.equals(svgDiagramConfig.getCacheEnabled())) {
            ImageData cached = tryReadCache(normalizedRequirement);
            if (cached != null) {
                log.info("SVG 缓存命中, cost={}ms", System.currentTimeMillis() - start);
                return cached;
            }
        }

        String svgCode = callLlmToGenerateSvg(normalizedRequirement);
        if (StrUtil.isBlank(svgCode)) {
            log.error("LLM 未生成 SVG 代码");
            return null;
        }

        if (!isValidSvg(svgCode)) {
            log.error("生成的 SVG 代码格式无效");
            return null;
        }

        if (Boolean.TRUE.equals(svgDiagramConfig.getCacheEnabled())) {
            writeCache(normalizedRequirement, svgCode);
        }

        byte[] svgBytes = svgCode.getBytes(StandardCharsets.UTF_8);
        log.info("SVG 概念示意图生成成功, size={} bytes, cost={}ms",
                svgBytes.length, System.currentTimeMillis() - start);
        return ImageData.fromBytes(svgBytes, "image/svg+xml");

    } catch (Exception e) {
        log.error("SVG 概念示意图生成异常, requirement={}", requirement, e);
        return null;
    }
}
```

### 5.6 tryReadCache 推荐行为

建议 tryReadCache(...) 的逻辑按这个顺序：

1. cacheEnabled=false 时直接返回 null
2. 算出 cache key 和 cache file path
3. 文件不存在 -> miss
4. 文件存在但已过期 -> 删除并 miss
5. 读出 SVG 文本
6. 再做一次 isValidSvg 校验
7. 校验通过 -> 返回 ImageData.fromBytes(...)
8. 校验失败 -> 删除坏缓存并 miss

建议方法签名：

```java
private ImageData tryReadCache(String normalizedRequirement)
```

### 5.7 writeCache 推荐行为

建议 writeCache(...) 做到这几点：

- 自动创建缓存目录
- 先写临时文件，再 move 覆盖目标文件
- 只缓存通过 isValidSvg 校验的结果

推荐结构：

```java
private void writeCache(String normalizedRequirement, String svgCode)
```

内部建议：

1. 计算 cache key
2. 解析 cacheDir
3. Files.createDirectories(...)
4. 先写 `${key}.tmp`
5. 再 move 成 `${key}.svg`

这样可以避免进程中断时留下半截文件。

## 6. 推荐你额外补的几个私有方法

为了让 SvgDiagramService 不变成一个大泥球，建议你把下面这些逻辑拆开：

- resolveCacheRoot()
- resolveCacheFile(String normalizedRequirement)
- isCacheExpired(Path cacheFile)
- deleteQuietly(Path file)

建议职责：

### 6.1 resolveCacheRoot

把 svgDiagramConfig.getCacheDir() 解析成绝对路径 Path。

### 6.2 resolveCacheFile

把 cache root + cache key 拼成最终缓存文件路径。

### 6.3 isCacheExpired

根据 Files.getLastModifiedTime(...) 和 cacheTtlMinutes 判断是否过期。

### 6.4 deleteQuietly

读到坏缓存或过期缓存时删文件，但删除失败不能影响主流程。

## 7. 你实现时最容易踩的坑

### 7.1 不要缓存未校验的 SVG

一定要在 isValidSvg(svgCode) 通过之后再写缓存。

否则一旦缓存里写进去坏 SVG，后面每次命中都会直接复用坏结果。

### 7.2 不要缓存最终访问 URL

当前第一版缓存不要缓存：

- upload 后的最终 URL
- 静态资源访问地址

因为这样会把缓存生命周期和上传文件生命周期强行耦合在一起。

第一版最稳的方式是：

- 缓存 SVG 内容
- 命中后重新返回 ImageData.fromBytes(...)
- 让 LocalImageStorageService 继续负责最终落盘

### 7.3 命中缓存后仍会生成一个新的上传文件，这是允许的

这是第一版最容易被误解的点。

缓存命中时：

- 你确实绕过了 LLM
- 但 LocalImageStorageService 仍会把这份 SVG 字节再落一次盘

这会带来重复文件，但这不是第一阶段的核心问题。

第一阶段只解决：

- 大模型耗时
- 重复生成耗时

不要为了“复用同一个最终 URL”把改动面扩大到整个图片存储链路。

### 7.4 缓存异常不能影响主流程

任何缓存相关失败都必须退化成“当次不走缓存、直接调 LLM”。

不能因为：

- 路径不存在
- 文件读写失败
- 计算 key 异常
- 缓存文件坏掉

就把整条 SVG 生成链路打断。

### 7.5 cacheDir 不要放到 static/uploads 下面

缓存目录建议和最终对外访问目录分开。

不要把缓存直接放进：

- src/main/resources/static/uploads

因为缓存不是对外资源，不需要暴露成静态访问路径。

## 8. 建议你手动实现时的顺序

建议按这个顺序手动改：

1. 先给 SvgDiagramConfig 和 application.yaml 加缓存字段
2. 再在 SvgDiagramService 里补 normalizeRequirement 和 buildCacheKey
3. 再补 resolveCacheFile、isCacheExpired、tryReadCache、writeCache
4. 最后改 generateSvgDiagramData 主流程
5. 编译通过后，再用重复输入做实际命中验证

这样比一口气全改更稳。

## 9. 验证方式

### 9.1 编译验证

先跑：

```bash
mvn -q -DskipTests compile
```

目标：

- 新配置字段能正常绑定
- SvgDiagramService 的 Path / Files / MessageDigest 相关代码编译通过
- 没有因为新加私有方法破坏现有接口实现

### 9.2 功能验证

固定一组只走 SVG 的输入，连续跑 2 次或 3 次。

建议输入：

- 主题：对日开发转国内开发
- 风格：情感风格
- 配图方式：仅 SVG

重点看：

- 第一次运行：应为 cache miss，仍会调 LLM
- 第二次运行：相同 requirement 应出现 cache hit
- 第二次 SVG 阶段耗时应明显下降

### 9.3 日志建议

建议至少补下面几类日志：

- SVG 缓存命中：key、耗时
- SVG 缓存未命中：key
- SVG 缓存过期：key
- SVG 缓存写入成功：key、文件路径
- SVG 缓存读取失败：key、异常
- SVG 总耗时：hit/miss、最终耗时

## 10. 预期结果

如果这次缓存改造完成，理想结果应该是：

- 相同或高度相似的 SVG 需求不会重复调用大模型
- 第二次及后续同类任务的 SVG 阶段耗时明显下降
- 缓存异常不会拖垮主链路
- 这次缓存改造不会阻塞你后面继续做模板快路径或结构化渲染
