# AI-Passage-Creator SVG 生成速度优化方案

## 1. 背景

在 taskId=ff24132c17944c109c88f849a679ee3c 的真实运行中，5 张 SVG 图片从开始生成到全部完成耗时约 327 秒，是整条文章创作链路中最主要的性能瓶颈。

当前图片链路中，SVG 的生成路径为：

1. ImageAnalyzerAgent 生成配图需求
2. ParallelImageGenerator 按 imageSource 分组
3. ImageGenerationTool.generateImageDirect 调用 ImageServiceStrategy
4. ImageServiceStrategy 调用 SvgDiagramService
5. SvgDiagramService 每张图单独调用一次 LLM 生成 SVG XML
6. LocalImageStorageService 落盘并返回本地 URL

## 2. 当前性能瓶颈拆解

### 2.1 同一来源内部串行执行

ParallelImageGenerator 当前策略是：

- 不同 imageSource 之间并行
- 同一 imageSource 内部串行

这意味着当 5 张图片全是 SVG_DIAGRAM 时，虽然代码叫“并行图片生成器”，实际执行效果是“1 组串行 SVG 任务”。

### 2.2 每张 SVG 都单独调用一次 LLM

SvgDiagramService 当前每张图都会单独构建 prompt，并调用一次大模型生成 SVG XML。这个调用本身是最重的步骤。

### 2.3 fallback 策略放大了 SVG 数量

当前只要模型返回了不被允许的来源，系统就直接使用 enabledMethods.getFirst() 作为 fallback。本次任务中，这使得多个原本非 SVG 的需求都变成了 SVG_DIAGRAM。

因此，SVG 慢不仅是 SvgDiagramService 自身的问题，还和上游“把太多图片导向 SVG”有关。

### 2.4 没有缓存

当前对于相似的 SVG prompt 没有缓存，也没有模板复用。即使多次生成类似结构图，也会重复调模型。

## 3. 优化目标

建议把优化目标拆成三档：

### 第一档：短期目标

- 5 张 SVG 的总耗时从 327 秒降到 120 秒以内
- 不改变现有业务结构，不推翻现有多 Agent 设计

### 第二档：中期目标

- 让“多来源图片选择”真正生效，避免所有图片都退化成 SVG
- 对高重复需求建立缓存和模板机制

### 第三档：长期目标

- 将“SVG 概念图”从纯 LLM 生成转向“结构化描述 + 本地渲染”模式，降低单次生成成本和不稳定性

## 4. 优化方案总览

## 方案 A：先减少 SVG 的数量

### 核心思路

不要让所有不合法来源都落到 SVG_DIAGRAM；优先恢复“多来源混合配图”。

### 具体做法

1. 修改 ImageAnalyzerAgent 的 fallback 策略，不再固定使用 enabledMethods.getFirst()。
2. 对不同图片类型做更细的 fallback：

- cover 优先 SVG_DIAGRAM 或 EMOJI_PACK
- section 配图优先 EMOJI_PACK
- 只有确实需要结构图时才回退到 SVG_DIAGRAM

3. 在 Prompt 中进一步约束：当允许方式只包含 EMOJI_PACK 和 SVG_DIAGRAM 时，普通章节优先 EMOJI_PACK，只有“关系图、概念图、流程图”才使用 SVG_DIAGRAM。

### 预期收益

- 直接减少需要走 SvgDiagramService 的图片数量
- 恢复不同来源分组并行的效果
- 用户选择“表情包 + SVG”时，最终结果更符合预期

### 推荐优先级

- P1

## 方案 B：允许同一来源内部有限并发

### 核心思路

当前同一来源内部完全串行过于保守。对于 SVG 这种纯生成型任务，可以放宽为“限流并发”。

### 具体做法

1. 修改 ParallelImageGenerator。
2. 保留“不同来源分组”的大结构不变。
3. 在单个来源内部，不再 for 循环串行，而是改为：

- 每个来源内部用 CompletableFuture 并发执行
- 加一个并发上限，例如 2 或 3
- 对 SVG_DIAGRAM 单独设置较低并发，避免把模型接口压满

### 预期收益

- 如果 5 张图都是 SVG，理论上可从完全串行优化为 2 路或 3 路并行
- 在不改业务结构的前提下，属于收益最高的提速方案之一

### 风险

- 并发过高可能导致模型 API 限流
- 本地存储、线程池和日志顺序会更复杂

### 推荐优先级

- P1

## 方案 C：增加 SVG 生成缓存

### 核心思路

对重复或高相似 prompt 生成结果做缓存，避免反复调 LLM。

### 具体做法

1. 在 SvgDiagramService 入口处，对 requirement 做规范化。
2. 计算 hash，例如：

- md5(requirement)

3. 先查询缓存：

- 可先用本地文件缓存
- 后续可迁移为 Redis 缓存或数据库缓存表

4. 命中缓存时直接返回 SVG 文件数据或本地 URL，不再调用 chatModel。

### 预期收益

- 对重复主题、重复结构图收益非常明显
- 对演示环境、回归测试环境特别有用

### 风险

- 需要处理缓存失效与文件清理
- 如果 requirement 文本稍有变化，缓存命中率可能不高

### 推荐优先级

- P1

## 方案 D：改成“结构化描述 + 本地 SVG 渲染”

### 核心思路

让模型只输出结构化 JSON 或 DSL，不直接输出完整 SVG XML；真正的 SVG 由本地代码渲染。

### 当前痛点

现在模型要一次性完成：

- 理解需求
- 设计结构
- 输出完整 SVG XML

这导致：

- 输出长
- token 多
- 校验复杂
- 单次调用慢

### 优化方向

改为两步：

1. LLM 只输出结构化描述，例如：

- 图类型：流程、关系、概念、对比
- 节点列表
- 连线关系
- 标题、说明文本

2. 本地 Java 根据结构化描述渲染 SVG。

### 预期收益

- 显著降低 LLM 输出长度
- 降低模型不稳定性
- 生成速度更快
- 后续更容易做统一风格和模板化

### 风险

- 需要补一套本地 SVG 渲染逻辑
- 开发量高于简单并发优化

### 推荐优先级

- P2

## 方案 E：为常见 SVG 场景做模板库

### 核心思路

不是所有 SVG 都需要 AI 现场生成。很多文章场景其实只需要固定结构模板。

### 适合模板化的场景

- 对比图
- 三阶段成长图
- 因果关系图
- 时间线
- 路径选择图
- 思维导图样式图

### 具体做法

1. 先抽 5 到 10 个高频模板。
2. ImageAnalyzerAgent 输出 prompt 时增加 templateType。
3. 如果命中模板，直接本地填充模板并生成 SVG。
4. 未命中模板时再退回到 LLM 生成。

### 预期收益

- 高频场景生成速度可大幅提升
- 稳定性比纯 LLM SVG 更高

### 推荐优先级

- P2

## 5. 推荐实施顺序

### 第 1 步：先控量

先优化 ImageAnalyzerAgent 的 fallback 和 Prompt 约束，减少 SVG 数量。

理由：

- 这是投入最小、收益直接的方案
- 能同时修复“选了表情包 + SVG，结果全变 SVG”的体验问题

### 第 2 步：再提并发

在 ParallelImageGenerator 中对 SVG_DIAGRAM 增加有限并发，建议从并发数 2 开始。

理由：

- 可以显著压缩 5 张 SVG 全部串行时的等待时间
- 对当前架构侵入较小

### 第 3 步：补缓存

为 SvgDiagramService 增加 requirement 级别缓存。

理由：

- 对重复场景收益大
- 可作为后续演示、回归跑流程的提速手段

### 第 4 步：再考虑架构升级

如果后续要把 SVG 作为长期能力，再推进“结构化描述 + 本地渲染”或模板库。

## 6. 建议修改点

### 代码改动候选

- ImageAnalyzerAgent：调整 fallback 逻辑与 Prompt 约束
- ParallelImageGenerator：增加同源限流并发
- SvgDiagramService：增加缓存与监控埋点
- 配置文件：增加 SVG 并发数、缓存开关、缓存 TTL

### 建议新增配置项

```yaml
svg-diagram:
  enabled: true
  max-concurrency: 2
  cache-enabled: true
  cache-ttl-minutes: 1440
  prompt-max-length: 800
```

## 7. 验证方案

每做一步优化，都建议固定一组测试输入进行对比：

- 主题：对日开发转国内开发
- 风格：情感风格
- 配图方式：表情包 + SVG

记录以下指标：

- 标题阶段耗时
- 大纲阶段耗时
- 正文阶段耗时
- 配图分析耗时
- SVG 总耗时
- 总链路耗时
- 最终不同图片来源的分布情况

## 8. 结论

当前 SVG 慢，不是单一服务实现差，而是多个因素叠加：

- 同源串行
- 每张图单独调 LLM
- 无缓存
- fallback 策略把过多图片压到 SVG

从投入产出比来看，最推荐的顺序是：

1. 先减少 SVG 数量
2. 再做 SVG 有限并发
3. 再补缓存
4. 最后考虑结构化渲染和模板库

这样既能贴近当前实现，也最适合作为下一阶段的小步优化路线。
