项目名称：AI-Passage-Creator

文档目的：
把当前“基本流程跑通”的状态，继续提升到“可写进简历、可支撑 1 年经验级别面试追问”的状态。

----------------------------------------------------------------------------

一、项目当前定位

该项目是三项目中的第一主项目，主要承担以下职责：

1. 体现 AI 应用开发能力。
2. 体现多 Agent 编排、交互式创作和 SSE 流式反馈能力。
3. 体现你不是只会接模型接口，而是能把 AI 能力组织成一个完整业务系统。

当前状态判断：

1. 功能链路已经完整。
2. 可以写进简历。
3. 还需要补工程化说明、架构取舍材料、演示素材和抗追问能力。

----------------------------------------------------------------------------

二、补强目标

补强完成后，应达到以下状态：

1. 面试时能在 3 分钟内讲清项目整体链路。
2. 面试官追问多 Agent、双引擎、SSE、配图容错时能稳定回答。
3. 项目仓库有对外可读的 README 和快速启动说明。
4. 有关键场景材料和业务闭环展示，不只是口头说“功能跑通了”。

----------------------------------------------------------------------------

三、优先级最高的补强项

3.1 README 和快速启动说明

必须补：

1. 项目简介。
2. 技术栈。
3. 系统架构或多 Agent 链路说明。
4. 核心功能列表。
5. 本地启动方式。
6. 配置项说明。
7. 数据库初始化方式。
8. 演示截图。

目的：

1. 让项目看起来像一个完整交付物。
2. 让面试官或其他人能快速理解这个项目不是 Demo。

3.2 业务闭环说明与演示材料

至少整理一份完整业务闭环说明，建议覆盖：

1. 输入选题。
2. 生成标题方案。
3. 用户确认标题。
4. 生成大纲。
5. 人工编辑大纲。
6. AI 修改大纲。
7. 生成正文。
8. 配图需求分析。
9. 多来源配图。
10. 图文合成。
11. 文章落库。
12. 详情查看和 Markdown 导出。

材料建议：

1. 页面截图。
2. 关键日志。
3. 成功结果说明。
4. 每一步为什么这样设计。

重点不是补测试，而是让面试官快速相信这是一个完整 AI 应用，不是一次性调用大模型接口。

3.3 双引擎与可观测性说明

必须补清楚以下问题：

1. 为什么保留双引擎。
2. 默认为什么走 Orchestrator。
3. 旧版串行路径的价值是什么。
4. SSE 实时消息和 agent_log 持久化日志分别解决什么问题。
5. 两种可观测方式如何互补。

这是面试时最容易被追问的点之一。

----------------------------------------------------------------------------

四、中优先级补强项

4.1 JSON 容错和失败降级说明

要能说清：

1. 标题和配图需求为什么需要 JSON 容错解析。
2. 容错解析解决了什么工程问题。
3. 配图来源为什么要做允许列表约束。
4. 图片服务失败时为什么要降级到 Picsum。

4.2 前端工程化补充

检查并确认以下内容：

1. 前端构建脚本是否稳定。
2. TypeScript 类型检查是否能通过。
3. 页面状态切换是否清晰。
4. SSE 断流或异常时前端是否有合理提示。

如果还有零散类型问题或临时实现，需要顺手收掉。

4.3 截图与演示材料整理

建议整理以下截图：

1. 创作首页。
2. 标题确认页。
3. 大纲编辑页。
4. 正文流式生成中。
5. 多图进度展示。
6. 创作完成详情页。
7. 后台用户管理页。

----------------------------------------------------------------------------

五、低优先级但高加分项

1. 给 Orchestrator 主链路补更统一的执行统计。
2. 增加简单自检脚本或关键链路验证脚本。
3. 补一份“后续可优化方向”文档，例如模板管理、审核流、成本统计。

----------------------------------------------------------------------------

六、面试必须准备的高频问题

至少要能稳定回答：

1. 为什么拆成多个 Agent，而不是一次性生成全文。
2. 为什么保留双引擎。
3. SSE 为什么适合这个场景。
4. 用户为什么要在标题和大纲阶段确认。
5. JSON 容错解析解决了什么问题。
6. 多来源配图为什么要按来源分组并行。
7. 允许列表约束和运行时降级分别解决什么问题。
8. 这个项目更像 Demo 还是业务系统。

----------------------------------------------------------------------------

七、完成标准

满足以下条件时，可认为 AI-Passage-Creator 已达到“简历级 + 一年经验级”标准：

1. 有完整 README。
2. 有快速启动说明。
3. 有完整业务闭环说明和演示材料。
4. 有关键页面截图。
5. 你能稳定讲清双引擎、多 Agent、SSE、配图容错、内容沉淀。
6. 面试高频问题能回答到位，不会停留在“因为这样更好”这种空话上。
