> 文档定位：项目技术选型说明 + 已实现功能落地总结

---

## 1. 项目概述

AI-Passage-Creator 是一个围绕“AI 文章创作”构建的全栈应用，整体形态不是简单的内容管理系统，而是一个 **以异步任务和人机协作创作为核心的内容生产平台**。

项目的核心目标有三层：

1. 让用户可以从一个选题快速进入文章创作流程。
2. 让 AI 生成过程透明可见，而不是用户提交后长时间黑盒等待。
3. 让最终产物可沉淀、可复查、可导出、可管理，而不是一次性结果。

从当前实现看，项目已经具备以下业务特征：

- 支持用户注册、登录、会话维持和管理员权限控制。
- 支持文章从创建任务到完成落库的完整闭环。
- 支持标题确认、大纲编辑、AI 修改大纲的人机协作式创作。
- 支持正文流式输出、配图分析、多来源配图与图文合成。
- 支持历史文章查看、详情展示、Markdown 导出和后台用户管理。

## 2. 项目总体架构

系统采用“前后端分离 + 异步任务编排 + SSE 实时推送”的实现方式。

### 2.1 前端架构定位

前端基于 Vue 3 构建，职责主要包括：

- 提供登录、注册、创作、文章列表、文章详情、管理员用户管理等页面。
- 在文章创作时承担阶段切换、流式渲染、交互确认和本地状态展示。
- 通过 Pinia 管理登录态，通过 Vue Router 实现路由装配和权限守卫。
- 通过 Axios 调用后端 API，通过 EventSource 接收文章生成进度。

### 2.2 后端架构定位

后端基于 Spring Boot 构建，职责主要包括：

- 提供用户、文章、SSE 等 API 接口。
- 维护用户登录态和文章任务生命周期。
- 编排大模型调用流程，驱动标题、大纲、正文、配图分析与图片生成。
- 负责图片下载、落盘、本地静态资源映射和异常降级。
- 持久化用户数据、文章产物与任务状态。

### 2.3 架构特点

当前架构最有辨识度的地方在于以下三点：

1. **不是一次性生成，而是分阶段交互式生成**。
2. **不是同步阻塞式响应，而是异步任务 + SSE 流式回传**。
3. **不是只生成文本，而是完整图文文章产物落库和展示**。

## 3. 技术选型总结

## 3.1 后端技术选型

### Java 21

后端语言选择 Java 21，主要价值在于：

- 生态成熟，适合构建稳定的企业级后端服务。
- 与 Spring Boot 3.x 兼容良好，适合项目长期演进。
- 对多线程、HTTP 调用、时间处理等基础能力支持完整。

### Spring Boot 3.5.10

项目使用 Spring Boot 作为后端主框架，主要用于：

- 快速搭建 Web API 服务。
- 统一管理 IOC 容器、配置注入和 Bean 生命周期。
- 结合注解式开发降低样板代码量。
- 便于接入 AOP、Redis Session、OpenAPI 等周边能力。

该技术选型适合本项目，因为项目既需要常规 Web 服务，也需要较强的配置组织能力和可扩展性。

### MyBatis-Flex 1.11.1

数据访问层采用 MyBatis-Flex，而不是更偏重实体关系映射的 JPA。当前选择更偏向以下诉求：

- SQL 查询和分页控制更直接。
- 适合以“表结构 + 查询条件 +分页”为主的后台业务。
- 与当前 `user`、`article` 两张核心表的结构较为匹配。

在本项目里，它落地在：

- 用户分页管理。
- 文章分页查询。
- 根据 `taskId`、`userId`、`status` 等条件检索文章。

### MySQL

MySQL 作为主数据库，承担项目核心业务数据的持久化：

- 用户信息。
- 文章任务信息。
- 标题、大纲、正文、配图与完整图文内容。
- 状态、阶段、错误信息与创建完成时间。

它适合当前项目的原因是：

- 业务结构化数据清晰，关系简单。
- 需要稳定持久化和常规分页查询能力。
- 方便后续继续扩展管理后台与统计功能。

### Redis + Spring Session Redis

项目没有使用 JWT，而是采用基于 Session 的登录态方案，并把 Session 存入 Redis。

这一方案的落地点包括：

- 登录后用户状态可跨请求维持。
- 前端只需携带 Cookie，无需自行管理 Token 续期逻辑。
- 管理员权限校验、文章权限校验都可以直接从 Session 中获取当前登录用户。

这对于当前以 Web 管理台和浏览器访问为主的系统是务实选择。

### Spring AOP

项目通过 AOP 实现 `@AuthCheck` 注解式权限控制，目的不是做复杂安全框架替代，而是用更轻量的方式解决：

- 管理员接口限制。
- 普通用户接口登录校验。
- 控制器层的统一权限拦截。

这使权限逻辑不必散落在每个方法中，代码可读性更好。

### Spring AI Alibaba Agent Framework + DashScope

这是整个项目最核心的技术选型之一，承担 AI 生成能力。

其在项目中的实际落地包括：

- 标题方案生成。
- 大纲流式生成。
- 正文流式生成。
- 配图需求分析。
- AI 修改大纲。
- SVG 图示生成等 AI 驱动场景。

之所以适合本项目，是因为系统不是只做一次简单问答，而是要把大模型能力拆进多个业务步骤中，形成稳定流程。

### Gson

Gson 在项目中的作用非常明确：

- 解析 LLM 输出的 JSON。
- 持久化大纲、标题方案、图片列表等结构化字段。
- 在实体、VO、SSE 消息之间做序列化转换。

由于 AI 输出常带有 JSON 格式结果，Gson 在这里是非常直接且高频的支撑工具。

### Jsoup

Jsoup 主要用于表情包抓取场景，其价值不在业务主链路中心位置，但属于多来源配图能力的重要补充。

### Mermaid CLI

Mermaid CLI 让项目不只依赖“图库图片”，而是支持流程图和结构图生成。这意味着文章配图可以不局限于照片，更适合科技、教程、分析类文章。

### Knife4j + SpringDoc OpenAPI3

这组技术为项目提供：

- 后端接口文档展示。
- 前端 SDK 自动生成基础。
- 更稳定的前后端协作方式。

它的价值不是“锦上添花”，而是降低前后端联调成本，尤其是在接口较多、状态流转较复杂的场景下非常实用。

## 3.2 前端技术选型

### Vue 3

项目选择 Vue 3 作为前端框架，适合当前业务的原因是：

- 组件化能力强，适合页面拆分。
- 响应式系统适合处理创作阶段状态切换。
- 与 TypeScript、Vite、Pinia、Vue Router 组合成熟。

项目中的落地页面包括：

- 首页。
- 登录页。
- 注册页。
- 文章创作页。
- 文章列表页。
- 文章详情页。
- 管理员用户管理页。

### Vite 8

Vite 作为构建工具的作用主要是：

- 提供更轻量的开发体验。
- 加快 Vue + TypeScript 项目的启动与构建。
- 简化前端工程配置。

### TypeScript 6

TypeScript 的使用不是装饰性的，而是实际参与到以下场景：

- API 请求类型约束。
- SSE 消息对象结构约束。
- 页面组件 props 与 emits 约束。
- 列表、详情、大纲结构等数据对象约束。

### Pinia

Pinia 主要用于登录用户状态管理，落地效果体现在：

- 全局统一维护当前登录用户。
- 路由守卫和页面都能共享登录态。
- 登录后刷新用户信息更加清晰。

### Vue Router

路由系统承载了项目的页面结构：

- `/`：首页。
- `/create`：文章创作页。
- `/article/list`：文章列表页。
- `/article/:taskId`：文章详情页。
- `/user/login`：登录页。
- `/user/register`：注册页。
- `/admin/userManage`：管理员页。

配合 `access.ts` 后，路由不仅负责页面切换，还承担权限入口职责。

### Axios

Axios 是项目统一的 HTTP 请求层，主要落地在：

- 统一 `baseURL`。
- 统一开启 `withCredentials` 以携带 Session Cookie。
- 全局拦截未登录状态并跳转登录页。

### EventSource / SSE

SSE 是本项目交互体验的关键技术之一。其落地不是简单展示日志，而是实实在在驱动创作页的状态变化：

- 标题方案生成完成。
- 大纲流式展示。
- 正文流式展示。
- 图片生成进度更新。
- 图文合成完成通知。

### Ant Design Vue

项目选择 Ant Design Vue 作为 UI 组件库，主要服务于：

- 表单。
- 表格。
- 弹窗。
- 按钮、标签、进度条。
- 管理后台交互。

这种选型更偏向“工程效率”和“管理台适配度”。

### marked

由于文章正文和完整图文以 Markdown 形式保存，前端必须有一个稳定的 Markdown 渲染方案，因此选择 marked 很自然。

### sortablejs

sortablejs 的加入，是交互式创作升级的重要标志之一。它让大纲编辑从“纯文本修改”升级成“结构化拖拽重排”，直接提升了用户参与感和编辑效率。

## 3.3 配置与工程支撑选型

当前工程层面的配置选型也比较清晰：

- `application.yaml` 负责集中配置。
- `application-local.yaml` 负责本地私密配置。

这说明项目不是只关注功能，也开始兼顾实际运行环境中的可部署性与可维护性。

## 4.2 文章创作主流程能力

这是项目最核心、也最有完成度的部分。

### 当前主流程

1. 用户输入选题。
2. 可选文章风格。
3. 可选允许的配图方式。
4. 创建文章任务。
5. 异步生成标题方案。
6. 用户确认标题并补充描述。
7. 生成文章大纲。
8. 用户编辑或 AI 修改大纲。
9. 用户确认大纲。
10. 生成正文。
11. 分析配图需求。
12. 生成或检索配图。
13. 图文合成。
14. 持久化保存并标记完成。

### 落地价值

相比传统“一键生成全文”的 AI 页面，这条流程更适合真实创作场景，因为：

- 用户对标题有参与权。
- 用户对大纲有修改权。
- AI 负责繁重生成，用户负责关键决策。
- 最终产物比纯自动生成更可控。

这意味着项目已经从“AI 帮写 demo”升级成“AI 协作创作工具”。

## 4.5 多来源配图能力

当前项目的配图体系并不是“给文章随机贴图”，而是已经形成了较完整的策略化实现。

### 已支持的配图来源

- Pexels：真实图片图库。
- Mermaid：流程图、结构图。
- Iconify：图标类资源。
- Emoji Pack：表情包类图片。
- SVG Diagram：AI 生成示意图。
- Picsum：降级兜底图片。

### 落地机制

- Agent 先分析正文中的配图需求。
- 为每张图生成位置、关键词或提示词。
- 策略层根据来源选择具体服务。
- 获取到的图片统一落盘到本地静态目录。
- 最终返回项目自身可访问 URL，而不是第三方原图直链。

### 落地价值

这使系统具备了以下能力：

- 科普、教程、分析型文章不只依赖照片，还可以生成图解内容。
- 多图片来源让不同文章风格的视觉表达更灵活。
- 本地落盘降低了第三方 URL 失效对详情页展示的影响。

## 5. 前后端协作方式总结

当前前后端协作采用的是较典型也较稳妥的模式：

### API 协作

- 后端通过 REST API 提供用户和文章能力。
- 前端通过 Axios 统一调用。
- OpenAPI 文档支持 SDK 生成，降低手写请求代码的不一致风险。

### 权限协作

- 前端通过路由守卫提前拦截管理员页面。
- 后端通过 Session + `@AuthCheck` 做最终安全校验。

### 状态协作

- 前端负责展示阶段。
- 后端负责维护真实状态和阶段。
- SSE 负责把后端任务执行过程同步到前端页面。

### 数据协作

- 后端以 `ArticleVO`、`LoginUserVO` 等对象对外输出。
- 前端依据结构化数据渲染页面，不直接绑定实体对象。

这套协作方式比较适合当前项目规模，既没有过度设计，也没有耦合得过于混乱。

## 6. 工程实现亮点

结合当前代码，项目有几个比较值得强调的亮点：

### 亮点 1：创作流程从线性生成升级为交互式状态机

这不是简单多加两个接口，而是把文章生成过程拆成了可确认、可暂停、可继续推进的阶段性工作流。

### 亮点 2：SSE 不只是日志推送，而是页面状态驱动机制

前端不是单纯显示消息，而是根据消息类型切换页面阶段、更新进度条、切换展示内容。

### 亮点 3：图片能力做成了策略层，而不是控制器里硬编码分支

`ImageServiceStrategy` 把图片来源扩展做成了可注册、可降级、可统一处理的方式，这对后续扩展新图片来源很重要。

### 亮点 4：文章产物不是临时变量，而是完整持久化资产

标题、大纲、正文、图片、完整图文、阶段、状态都被保存下来，使系统具备复查和再利用价值。

### 亮点 5：已经开始考虑真实网络环境问题

`ExtraTrustStoreInitializer` 的存在说明项目已经遇到并处理了 HTTPS 证书链问题，这比很多只停留在本地开发 happy path 的项目更进一步。
