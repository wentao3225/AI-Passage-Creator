# AI-Passage-Creator 项目技术选型与功能落地总结（精简版）

> 项目名称：AI-Passage-Creator  
> 文档定位：简明介绍项目技术方案与已实现功能  
> 基线版本：当前工作区代码（2026-04-22）

---

## 1. 项目简介

AI-Passage-Creator 是一个面向 AI 文章创作场景的前后端一体化系统。系统围绕“选题输入 -> AI 生成 -> 用户确认 -> 图文落库”构建完整闭环，不是单次问答式页面，而是一个 **多 Agent 协同创作系统**。

当前版本的核心特点是：**双引擎可切换编排（默认 Orchestrator）、交互式创作、SSE 流式反馈、容错解析、多来源并行配图、文章沉淀和后台管理**。

---

## 2. 技术选型概述

### 2.1 后端技术选型

后端采用 Java 21 + Spring Boot 3.5.10，负责 API、业务编排和系统运行。数据层使用 MyBatis-Flex + MySQL，登录态采用 Redis + Spring Session Redis，权限控制通过 Spring AOP + `@AuthCheck` 实现；旧版 Agent 服务路径与 AI 改纲路径可通过 `@AgentExecution` 采集执行日志。

AI 能力基于 Spring AI Alibaba Agent Framework + DashScope 落地，负责标题生成、大纲生成、正文生成、配图需求分析和 AI 修改大纲。当前默认走 `ArticleAgentOrchestrator`（StateGraph），保留 `ArticleAgentService` 作为回退路径。Gson 与 `LlmJsonUtils` 用于 LLM JSON 结构容错解析，Java HttpClient、Jsoup 和 Mermaid CLI 用于图片下载、表情包抓取和流程图类配图生成。

### 2.2 前端技术选型

前端采用 Vue 3 + TypeScript + Vite，负责创作流程、页面交互和结果展示。Pinia 用于登录态管理，Vue Router 用于路由与管理员守卫，Axios 用于统一请求，EventSource 用于接收 SSE 消息。Ant Design Vue 提供通用 UI，marked 负责 Markdown 渲染，sortablejs 用于大纲拖拽排序。

### 2.3 工程配置选型

项目通过 `application.yaml` 管理公共配置，通过 `application-local.yaml` 隔离本地敏感配置，并额外实现了异步线程池、本地图片静态资源映射和 truststore 初始化逻辑。通过 `article.agent.orchestrator.enabled` 控制双引擎切换，默认值为 `true`（启用编排器）。

## 3. 多 Agent 架构与实现思路

系统整体采用“前后端分离 + 异步任务 + SSE 实时推送 + 多 Agent 协同”的方式实现。

前端负责输入选题、展示阶段、接收实时消息、确认标题和编辑大纲；后端负责创建任务、维护状态、调度多 Agent 生成链路，并把最终图文文章落库。

当前文章生成链路可概括为 5 个 Agent + 1 个合并节点：

1. Agent1：生成标题方案。
2. Agent2：生成文章大纲。
3. Agent3：生成正文内容。
4. Agent4：分析配图需求。
5. Agent5：生成或检索配图。
6. merge：将配图占位符替换为最终图片并输出完整图文。

当前运行时为“可切换双引擎”：

1. `article.agent.orchestrator.enabled=true`：默认走 `ArticleAgentOrchestrator`（StateGraph），阶段 3 支持按图片来源分组并行配图。
2. `article.agent.orchestrator.enabled=false`：回退到 `ArticleAgentService` 串行实现。

在此基础上，系统把整条链路拆成“标题确认 -> 大纲确认 -> 正文完成”三个用户可参与阶段，使生成过程可见、可干预、可追踪。
同时，标题与配图需求节点已经接入 LLM JSON 容错解析，降低格式抖动导致的阶段失败。

## 4. 已实现功能落地总结

## 4.1 用户与权限功能

项目已经实现用户注册、登录、获取当前登录用户和登出。登录态通过 Redis Session 持久化，前端路由守卫和后端 `@AuthCheck` 共同完成权限控制。

管理员后台已支持用户分页查询、新增、编辑和删除，普通用户不能访问后台接口。

## 4.2 文章创作主流程功能

文章创作是系统核心业务。用户在创作页输入选题后，可选设置文章风格和允许的配图方式，提交后由后端异步启动多 Agent 生成流程。

当前已经实现的创作功能包括：

1. 根据选题生成多个标题方案。
2. 用户可以选择系统给出的标题，也可以自定义标题。
3. 用户可以填写补充描述，告诉 AI 更偏向哪些内容。
4. 系统根据确认后的标题生成文章大纲。
5. 用户可以手动编辑大纲。
6. 用户可以向 AI 提交修改建议，让 AI 重新调整大纲。
7. 用户确认大纲后，系统继续生成正文。
8. 正文生成完成后，系统自动分析哪些位置需要配图。
9. 系统继续执行图片生成、图片检索和图文合成。
10. 最终保存为完整文章记录，供后续查看和导出。
11. 根据配置开关在编排器链路与旧版服务链路之间切换执行。

这部分体现的是“多 Agent + 用户确认”的协作式创作，而不是一次性全文返回。

## 4.3 流式生成与实时反馈功能

系统已经实现 SSE 实时推送能力，前端会根据后端消息实时更新界面。

目前已经落地的实时反馈包括：

1. 标题方案生成完成通知。
2. 大纲流式输出。
3. 大纲生成完成并进入编辑阶段。
4. 正文流式输出。
5. 配图需求分析完成通知。
6. 单张图片生成完成通知。
7. 全部图片完成通知。
8. 图文合成完成通知。
9. 全流程完成通知。
10. 错误通知。

前端会据此切换页面阶段、更新实时日志和图片进度条，并最终进入“创作完成”状态。

补充说明：默认编排器链路下，过程可观测性以 SSE 实时消息为主；`agent_log` 持久化日志主要来自旧版服务链路与 AI 改纲调用。

## 4.4 人机协作编辑功能

这是当前版本的关键特点之一，系统让用户在中间阶段参与决策，而不是只接收最终结果。

目前已实现的人机协作能力包括：

1. 多标题方案选择。
2. 自定义主标题和副标题。
3. 用户补充描述输入。
4. 大纲章节编辑。
5. 大纲要点编辑。
6. 大纲章节新增和删除。
7. 大纲要点新增和删除。
8. 大纲章节拖拽排序。
9. AI 修改大纲。

这让 AI 负责生成效率，用户负责方向控制。

## 4.5 多来源配图功能

项目已经实现多来源配图体系，而不是只依赖一个图片接口。目前系统支持以下配图来源：

1. Pexels 真实图片图库。
2. Mermaid 流程图或结构图。
3. Iconify 图标资源。
4. Emoji Pack 表情包图片。
5. SVG Diagram AI 生成示意图。
6. Picsum 作为失败时的降级兜底。

系统会先由 AI 分析正文中的配图需求，再为每张图片生成对应的关键词或提示词，然后调用具体图片服务获取图片。在默认编排器链路下，配图会按来源分组并行执行，提升多图场景吞吐。

另外，配图环节已经实现两层约束：

1. 允许列表约束：模型返回的来源若不在用户允许列表中，会被后端强制收敛。
2. 运行时降级约束：图片服务不可用或落盘失败时自动降级到 `PICSUM`。

图片获取成功后不会直接依赖第三方原始链接，而是统一保存到项目本地静态目录，再通过后端 `/uploads/**` 对外访问。

这使文章能够自动完成图文结合，并适配不同类型内容的图片表达。

## 4.6 文章管理与内容沉淀功能

系统已经具备完整的文章管理能力。文章不是生成完就消失，而是作为内容资产被保存下来。

目前已经实现的文章管理功能包括：

1. 查看文章详情。
2. 分页查看文章列表。
3. 文章删除。
4. 展示文章状态。
5. 展示当前阶段。
6. 展示配图数量。
7. 展示失败错误信息。
8. 详情页优先展示完整图文内容。
9. 详情页导出 Markdown。
10. 失败文章基于原选题重新创建。
11. 详情页自动拉取 `/article/execution-logs/{taskId}` 展示执行统计（若有持久化日志）。

这说明系统已经具备内容沉淀与再利用能力。

## 4.7 管理后台功能

除了用户侧功能，系统还已经实现管理员后台用户管理页面。管理员可以在后台按账号、昵称和角色筛选用户，并进行新增、编辑、删除等操作。

这说明项目已经具备基础治理能力。

## 4.8 执行可观测性与统计功能（条件生效）

项目已经落地 `@AgentExecution + AgentExecutionAspect + agent_log` 的执行日志采集能力，并提供 `/article/execution-logs/{taskId}` 聚合接口。

当前口径如下：

1. 旧版 `ArticleAgentService` 路径和 `ai-modify-outline` 调用会产生完整持久化执行日志。
2. 默认编排器主链路以 SSE 实时消息为主，详情页执行日志面板消费的是 `agent_log` 聚合结果。
3. 两类可观测性（实时与持久化）在职责上互补：前者用于过程感知，后者用于历史回看。

## 6. 项目总结

总体来看，AI-Passage-Creator 已经实现了一个较完整的 AI 内容创作系统原型。它的重点不只是接入大模型，而是把 **双引擎可切换编排、用户中途确认、SSE 实时反馈、容错解析、多来源并行配图和文章沉淀** 组织成了一条完整业务链路。

当前版本已经完整覆盖用户、创作、协作编辑、流式反馈、可观测性、配图、文章管理和后台治理等主要模块，具备较强的展示性和继续产品化扩展的基础。
