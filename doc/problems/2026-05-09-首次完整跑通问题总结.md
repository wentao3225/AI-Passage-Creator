## 3. 本次暴露出的主要问题

### 3.1 选择了“表情包 + SVG”，最终却全部生成成了 SVG

#### 现象

日志中 ImageAnalyzerAgent 收到的允许方式为：

- [SVG_DIAGRAM, EMOJI_PACK]

但模型输出的部分配图需求来源为 PEXELS，随后被过滤并替换：

- position=2, imageSource=PEXELS -> fallback=SVG_DIAGRAM
- position=3, imageSource=PEXELS -> fallback=SVG_DIAGRAM
- position=4, imageSource=PEXELS -> fallback=SVG_DIAGRAM

最终 ParallelImageGenerator 按来源分组时只剩：

- {SVG_DIAGRAM=5}

#### 根因

当前实现里，ImageAnalyzerAgent 会先让模型自由生成配图需求；如果模型返回的 imageSource 不在允许列表里，则不会重新做“二次择优”，而是直接使用允许列表中的第一个来源作为 fallback：

```java
String fallbackSource = enabledMethods.getFirst();
req.setImageSource(fallbackSource);
```

因此，只要模型给了多个不被允许的来源，并且 enabledMethods 的第一个值是 SVG_DIAGRAM，那么这些需求都会被统一替换为 SVG_DIAGRAM。

#### 影响

- 图片风格会单一化
- 用户选择了多种图片方式，但最终体验上像只启用了 1 种
- 不同来源并行的设计优势会失效
- SVG 图片数量过多时，会进一步放大整体耗时问题

#### 当前判断

这不是“前端没传对”的 bug，而是当前 fallback 策略过于粗糙导致的结果。

### 3.2 最终 phase 没有完成态

#### 现象

本次运行前的实现中，阶段 3 完成后只会：

- 保存文章内容
- 更新 status=COMPLETED

但不会更新 phase，因此数据库里会出现类似状态：

- status=COMPLETED
- phase=CONTENT_GENERATING

#### 根因

ArticlePhaseEnum 中没有 COMPLETED 阶段；ArticleAsyncService.executePhase3 结束时也没有补写最终 phase。

#### 影响

- phase 字段不能准确表达任务生命周期终态
- 列表页或后台排查时容易误判为“还在生成正文”
- status 和 phase 表达不一致

#### 当前处理状态

该问题已在本次改动中修复：

- 新增 ArticlePhaseEnum.COMPLETED
- executePhase3 末尾补写 article phase=COMPLETED
- 前端列表页补充 COMPLETED 的阶段文案和样式

### 3.3 SVG 生成耗时过长，成为整条链路最慢环节

#### 现象

本次 5 张 SVG 从开始生成到全部完成大约耗时 5 分 27 秒，明显高于标题、大纲和正文阶段。

#### 根因

当前 SVG 生成路径是：

- 每张 SVG 都单独调用一次 LLM 生成 SVG XML
- 同一来源的图片在 ParallelImageGenerator 中是串行执行的
- 本次所有图片最终都变成了 SVG_DIAGRAM，因此虽然类名叫 ParallelImageGenerator，实际只剩 1 组串行任务

#### 影响

- 用户等待时间过长
- SSE 虽然能持续显示进度，但“完成感”较差
- 一旦文章章节较多、图片较多，整体体验会迅速恶化

#### 当前判断

这不是单点代码 bug，而是当前图片策略、fallback 策略和并发策略叠加后的性能问题。

### 3.4 ContentMergerAgent 对封面图打出了误导性警告

#### 现象

图文合成阶段出现日志：

- 图片 position=1 的 placeholderId 为空

#### 根因

根据 Prompt 设计，position=1 表示封面图，本来就不要求正文占位符；但 ContentMergerAgent 当前对所有图片统一判断，只要 placeholderId 为空就打 warn。

#### 影响

- 排查日志时容易把正常设计误判成异常
- 会干扰真正的占位符问题定位

#### 当前判断

这是日志语义不准确，不影响主流程功能。
